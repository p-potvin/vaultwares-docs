## Vue d'ensemble

Ce guide présente les meilleures pratiques de sécurité pour les déploiements VaultWares, basées sur NIST SP 800-57, les CIS Controls et les leçons tirées des déploiements de sécurité en entreprise. Le suivi de ces recommandations maximisera la valeur de sécurité de votre investissement VaultWares.

## Meilleures pratiques de gestion des clés

### Principe du moindre privilège pour les clés

<AccordionGroup>
  <Accordion title="Séparation des objectifs des clés">
    N'utilisez jamais la même clé cryptographique pour plusieurs objectifs. Maintenez des clés séparées pour :
    - Chiffrement des données (une clé par niveau de sensibilité, ex. : `PII-Data-Key`, `Financial-Data-Key`)
    - Chiffrement de sauvegarde (`Backup-Master-Key`)
    - Certificats TLS (`TLS-WebServer-Key`, `TLS-API-Key`)
    - Signature de code (`Code-Signing-Key`)
    - Signature des journaux d'audit (`AuditLog-Signing-Key`)
  </Accordion>
  <Accordion title="Hiérarchie des clés">
    Utilisez une hiérarchie de clés : Clés maîtresses (stockées dans l'HSM, jamais exportées) → Clés de chiffrement des données (DEKs) → Clés de session dérivées. Cela limite le rayon d'impact en cas de compromission d'une clé.

    ```bash
    # Generate a master key (never extractable)
    vw-hsm generate-key \
      --type aes \
      --size 256 \
      --label "Master-Key-v1" \
      --extractable false \
      --usage wrap,unwrap

    # Generate a DEK protected by the master key
    vw-hsm generate-key \
      --type aes \
      --size 256 \
      --label "PII-DEK-v1" \
      --extractable true \
      --wrap-with Master-Key-v1
    ```
  </Accordion>
  <Accordion title="Gardiennage des clés M-sur-N">
    Pour les clés maîtresses, utilisez toujours le fractionnement de clés M-sur-N. Aucune personne seule ne devrait être capable de récupérer une clé maîtresse. Un fractionnement 3-sur-5 est recommandé pour la plupart des organisations.

    ```bash
    vw-hsm backup create \
      --key Master-Key-v1 \
      --split 3-of-5 \
      --custodians alice,bob,charlie,diana,erin \
      --output /secure/backup/master-key-backup.enc
    ```
  </Accordion>
</AccordionGroup>

### Calendrier de rotation des clés

| Type de clé | Rotation recommandée | Notes |
|----------|---------------------|-------|
| Certificats TLS (visibles publiquement) | Annuelle ou en cas de compromission | Utiliser des certificats à courte durée (90 jours) avec ACME si possible |
| Certificats TLS (internes) | 2 ans | Automatiser avec ACME ou SCEP interne |
| Clés de chiffrement des données | Annuelle | Re-chiffrer les données avec une nouvelle DEK lors de la rotation |
| Clés de chiffrement de sauvegarde | Annuelle | Conserver les anciennes clés pour le déchiffrement de sauvegarde |
| Clés d'API | 90 jours | Automatiser la rotation si possible |
| Clés maîtresses HSM | 3–5 ans | Seulement avec une cérémonie de clés complète |
| Mots de passe administrateur | 90 jours | Imposer via la politique de mot de passe VaultAccess |

```bash
# Check upcoming key expirations
vw-hsm key list --expiring-within 60d

# Schedule automated key rotation
vw-admin key-rotation schedule \
  --key-pattern "TLS-*" \
  --interval 365d \
  --notification-lead 30d \
  --alert-email security@example.com
```

## Politiques de mot de passe

### Politique de mot de passe recommandée

```json
{
  "password_policy": {
    "min_length": 16,
    "require_uppercase": true,
    "require_lowercase": true,
    "require_numbers": true,
    "require_special_chars": true,
    "disallow_common_passwords": true,
    "disallow_username_in_password": true,
    "max_age_days": 90,
    "history_count": 24,
    "lockout_threshold": 5,
    "lockout_duration_minutes": 30,
    "admin_unlock_required": false
  }
}
```

<Tip>
  Pour les appareils VaultDrive, envisagez de définir une longueur de mot de passe minimale de 12 caractères, avec un verrouillage après 10 tentatives infructueuses et un effacement cryptographique déclenché après 15 échecs consécutifs.
</Tip>

### Mots de passe des comptes à privilèges

```bash
# Enforce stricter policy for privileged accounts
vw-admin policy create \
  --name PrivilegedAccountPolicy \
  --min-length 20 \
  --require-all-character-classes true \
  --max-age-days 60 \
  --require-mfa hardware-key \
  --apply-to-roles Administrator,SecurityOfficer,CryptoAdmin
```

## Sécurité réseau

### Segmenter votre infrastructure de sécurité

<AccordionGroup>
  <Accordion title="Isolation réseau de l'HSM">
    Les appareils VaultHSM doivent être sur un VLAN dédié, accessible uniquement à partir des serveurs d'applications qui nécessitent des services cryptographiques. Ne jamais exposer les interfaces de gestion de l'HSM aux réseaux d'entreprise généraux.

    ```bash
    # Example firewall rule: allow only app servers to reach HSM
    # Source: 10.10.20.0/24 (App Server VLAN)
    # Destination: 10.10.30.10 (HSM VLAN)
    # Port: TCP 1792 (PKCS#11 over TLS)
    # Allow. Log all connections.
    ```
  </Accordion>
  <Accordion title="Réseau de gestion">
    Toutes les interfaces de gestion VaultWares (consoles, SSH) doivent être sur un réseau de gestion dédié et hors bande, accessible uniquement via VPN ou serveur de saut avec MFA.
  </Accordion>
  <Accordion title="Segmentation VaultGate">
    Déployer VaultGate aux frontières réseau pour appliquer une communication chiffrée entre les segments. Tout le trafic inter-segment doit être authentifié et chiffré.
  </Accordion>
</AccordionGroup>

### Durcissement TLS

```bash
# Configure minimum TLS 1.2 and strong cipher suites
vw-admin tls configure \
  --min-version TLSv1.2 \
  --ciphers "TLS_AES_256_GCM_SHA384:TLS_CHACHA20_POLY1305_SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384" \
  --hsts-max-age 31536000 \
  --hsts-include-subdomains true \
  --hsts-preload true

# Disable legacy protocols on VaultGate
vaultgate ssl-policy set \
  --min-version TLSv1.2 \
  --disable-weak-ciphers true \
  --disable-compression true
```

## Stratégie de sauvegarde

### La règle 3-2-1-1

Suivez la stratégie de sauvegarde élargie 3-2-1-1 pour les données critiques :

| Exigence | Implémentation |
|-------------|---------------|
| **3** copies de données | Original + sauvegarde locale + sauvegarde hors site |
| **2** types de médias différents | NAS sur site + stockage cloud |
| **1** copie hors site | Sauvegarde dans le cloud ou en co-localisation avec VaultBackup |
| **1** copie immuable | Référentiel immuable VaultBackup (protection contre les rançongiciels) |

```bash
# Verify all three backup destinations are healthy
vw-backup repo status --all

# Test restore quarterly (automate where possible)
vw-backup restore-test \
  --repo backup://nas01/vaults/prod \
  --sample-size 10 \
  --target /data/restore-test \
  --verify-checksums
```

## Réponse aux incidents

### Guide de réponse immédiate

<Steps>
  <Step title="Contenir l'incident">
    ```bash
    # Lock a compromised device
    vw-storage remote-lock --device VD-ENT-00247

    # Revoke a compromised user's access
    vw-admin user disable --user compromised-user@example.com
    vw-admin user revoke-sessions --user compromised-user@example.com

    # Isolate a compromised system via VaultGate
    vaultgate quarantine-host --ip 10.10.20.45 --reason "Suspected compromise"
    ```
  </Step>
  <Step title="Préserver les preuves">
    ```bash
    # Export audit logs for the affected user/device (do this before any remediation)
    vw-admin audit export \
      --user compromised-user@example.com \
      --start "2024-11-01T00:00:00Z" \
      --end "2024-11-15T23:59:59Z" \
      --output /forensics/incident-$(date +%Y%m%d)/user-audit.json \
      --sign-with hsm:AuditLog-Signing-Key

    vw-monitor export \
      --device VD-ENT-00247 \
      --all-events \
      --output /forensics/incident-$(date +%Y%m%d)/device-events.json
    ```
  </Step>
  <Step title="Évaluer la portée">
    ```bash
    # Check what resources the compromised account accessed
    vw-admin audit query \
      --user compromised-user@example.com \
      --since 30d \
      --group-by resource,operation \
      --format table
    ```
  </Step>
  <Step title="Remédier">
    Si des clés cryptographiques ont pu être exposées, initier une rotation de clés d'urgence :

    ```bash
    # Emergency key rotation
    vw-admin key-rotation emergency \
      --key-labels "DB-Master-Key,TLS-WebServer-Key" \
      --revoke-old true \
      --notify security@example.com \
      --incident-id INC-20241115-001
    ```
  </Step>
  <Step title="Documenter et réviser">
    Générer un rapport post-incident et mettre à jour les politiques en fonction des leçons apprises. Planifier un exercice de table (tabletop) dans les 30 jours.
  </Step>
</Steps>

## Liste de vérification de conformité

<AccordionGroup>
  <Accordion title="Contrôle d'accès">
    - [ ] MFA appliqué à tous les comptes utilisateurs
    - [ ] Clés matérielles requises pour les comptes à privilèges
    - [ ] Revues d'accès complétées (minimum trimestriel)
    - [ ] Comptes dormants révisés et désactivés (inactifs depuis 30+ jours)
    - [ ] RBAC de moindre privilège appliqué à tous les rôles
  </Accordion>
  <Accordion title="Gestion des clés">
    - [ ] Toutes les clés de chiffrement stockées dans VaultHSM (pas de coffres-forts logiciels)
    - [ ] Calendrier de rotation des clés documenté et automatisé
    - [ ] Sauvegarde M-sur-N en place pour les clés maîtresses
    - [ ] Gardiens des clés nommés et formés
    - [ ] Aucune clé partagée entre environnements (dev/staging/prod)
  </Accordion>
  <Accordion title="Audit et surveillance">
    - [ ] Journalisation d'audit activée sur tous les produits VaultWares
    - [ ] Journaux transférés au SIEM dans les 5 minutes
    - [ ] Règles d'alerte révisées et ajustées trimestriellement
    - [ ] Rétention des journaux conforme aux exigences réglementaires (1 à 7 ans)
    - [ ] Intégrité des journaux d'audit protégée (signature HSM activée)
  </Accordion>
  <Accordion title="Sauvegarde et récupération">
    - [ ] Stratégie de sauvegarde 3-2-1-1 implémentée
    - [ ] Clés de chiffrement de sauvegarde dans l'HSM
    - [ ] Récupération testée trimestriellement
    - [ ] Objectifs RTO/RPO documentés et vérifiés
    - [ ] Sauvegardes hors site vérifiées accessibles
  </Accordion>
</AccordionGroup>
