## Vue d'ensemble

La règle de sécurité HIPAA (45 CFR Part 164) exige des entités couvertes et des associés commerciaux qu'elles mettent en œuvre des sauvegardes administratives, physiques et techniques pour protéger les informations de santé protégées électroniques (ISPE). Les produits VaultWares répondent directement aux exigences de sauvegarde techniques.

<Note>
  Ce guide fournit des conseils techniques sur la manière dont les produits VaultWares soutiennent la conformité HIPAA. Consultez votre responsable de la conformité et votre conseiller juridique pour une analyse des risques HIPAA complète.
</Note>

## Exigences de sauvegarde techniques

La règle de sécurité HIPAA définit les sauvegardes techniques à 45 CFR § 164.312.

<Tabs>
  <Tab title="Contrôle d'accès (§164.312(a))">
    **Requis** : Identification unique de l'utilisateur, procédure d'accès d'urgence, déconnexion automatique, chiffrement/déchiffrement

    | Spécification | Statut | Contrôle VaultWares |
    |---|---|---|
    | Identifiants d'utilisateur uniques | Requis | Comptes utilisateurs uniques VaultAccess; pas de mots de passe partagés |
    | Accès d'urgence | Requis | Procédure de rupture de verre (break-glass) VaultAccess PAM |
    | Déconnexion automatique | Adressable | Délai de session inactive VaultAccess; Verrouillage automatique VaultDrive |
    | Chiffrement/déchiffrement | Adressable | Chiffrement de volume ISPE VaultCrypt; Gestion des clés VaultHSM |

    ```bash
    # Configure automatic session timeout for ePHI systems
    vw-admin policy create \
      --name HIPAA-ePHI-Access \
      --idle-timeout-minutes 15 \
      --require-mfa true \
      --require-hardware-key-for-privileged true \
      --apply-to-systems "ehr.corp.example.com,phi-db01.corp.example.com"

    # Configure VaultDrive auto-lock for clinical staff
    vw-storage set-policy \
      --policy clinical-staff \
      --auto-lock-minutes 5 \
      --require-reauthentication true
    ```
  </Tab>
  <Tab title="Contrôles d'audit (§164.312(b))">
    **Requis** : Mécanismes matériels, logiciels et procéduraux pour enregistrer et examiner l'activité dans les systèmes d'information contenant des ISPE

    ```bash
    # Enable comprehensive ePHI audit logging
    vw-admin audit configure \
      --scope ephi-systems \
      --log-level detailed \
      --capture-read-access true \
      --capture-write-access true \
      --capture-delete-access true \
      --capture-authentication-events true \
      --capture-failed-attempts true \
      --retention-days 2190 \  # 6 years (HIPAA minimum)
      --sign-with hsm:AuditLog-Signing-Key \
      --tamper-evident true

    # Generate audit report for a period
    vw-monitor report generate \
      --template hipaa-access-audit \
      --start "2024-10-01" \
      --end "2024-10-31" \
      --scope ephi \
      --format pdf \
      --output /reports/hipaa-audit-oct-2024.pdf
    ```

    Événements d'audit capturés par VaultMonitor pour HIPAA :

    | Événement | Niveau de détail |
    |---|---|
    | Accès à un fichier ISPE (lecture) | Utilisateur, horodatage, ressource, IP source |
    | Modification d'un fichier ISPE | Utilisateur, horodatage, ressource, résumé des changements |
    | Suppression / effacement crypto d'un fichier ISPE | Utilisateur, horodatage, ressource, raison |
    | Tentatives d'authentification | Utilisateur, horodatage, résultat, IP source, méthode MFA |
    | Sessions d'accès privilégié | Enregistrement complet de la session (PAM) |
    | Opérations de gestion des clés | Journal d'audit HSM — ID de clé, opération, utilisateur |
  </Tab>
  <Tab title="Intégrité (§164.312(c))">
    **Requis** : Mécanismes pour authentifier les ISPE; mécanismes électroniques pour corroborer que les ISPE n'ont pas été altérées ou détruites

    ```bash
    # Enable file integrity monitoring on ePHI directories
    vw-monitor integrity-monitor enable \
      --directories /data/phi,/var/lib/ehr \
      --hash-algorithm SHA-256 \
      --check-interval 15m \
      --alert-on-change true \
      --baseline-signed-with hsm:Integrity-Key

    # Verify ePHI backup integrity
    vw-backup verify-all \
      --repo backup://nas01/vaults/phi \
      --decrypt-key hsm:PHI-Backup-Key \
      --generate-integrity-report \
      --output /reports/phi-backup-integrity-$(date +%Y%m%d).json
    ```
  </Tab>
  <Tab title="Sécurité de transmission (§164.312(e))">
    **Requis** : Protection contre l'accès non autorisé aux ISPE transmises sur des réseaux électroniques

    ```bash
    # Enforce TLS 1.2+ for all ePHI transmission
    vaultgate policy create \
      --name HIPAA-ePHI-Transmission \
      --min-tls-version TLSv1.2 \
      --require-mutual-tls true \
      --allowed-cipher-suites "TLS_AES_256_GCM_SHA384,ECDHE-RSA-AES256-GCM-SHA384" \
      --deny-plaintext-for-hosts "ehr.corp.example.com,phi-db01.corp.example.com" \
      --alert-on-policy-violation true
    ```
  </Tab>
</Tabs>

## Configuration de protection des ISPE

### Chiffrer les ISPE au repos

```bash
# Create dedicated encrypted volume for ePHI
vaultcrypt create-container \
  --output /data/phi/phi-vault.vc \
  --size 500G \
  --algorithm AES-256-XTS \
  --key-source hsm \
  --key-label "PHI-Data-Key" \
  --require-2fa true \
  --auto-mount-on-boot true \
  --audit-all-access true

# Policy: encrypt all data written to ePHI directories
vaultcrypt policy set \
  --name PHI-Always-Encrypted \
  --paths /data/phi,/var/lib/ehr/attachments \
  --algorithm AES-256-XTS \
  --key-label "PHI-Data-Key" \
  --enforce-no-plaintext true
```

### Déploiement de VaultDrive pour le personnel clinique

```bash
# Deploy HIPAA-compliant policy to clinical staff VaultDrives
vw-admin deploy-config \
  --policy hipaa-clinical-staff \
  --device-group ClinicalStaff-VaultDrives \
  --settings '{
    "auto_lock_minutes": 5,
    "require_2fa": true,
    "require_pin_on_sleep": true,
    "audit_logging": "all",
    "allow_readonly_mode": true,
    "max_failed_attempts": 5,
    "crypto_erase_on_max_failures": false
  }'
```

## Accord d'associé commercial (AAC)

VaultWares exécute des Accords d'associé commercial avec les entités couvertes. Un AAC confirme que VaultWares :

- Protège la sécurité et la confidentialité des ISPE
- Signale les incidents de sécurité et les violations à l'entité couverte
- Retourne ou détruit les ISPE à la fin du contrat
- S'assure que les sous-traitants (y compris les fournisseurs de services cloud) exécutent des AAC

Contactez [compliance@vaultwares.com](mailto:compliance@vaultwares.com) pour demander un AAC.

## Liste de vérification de conformité HIPAA

<AccordionGroup>
  <Accordion title="Contrôle d'accès">
    - [ ] Identifiants d'utilisateur uniques appliqués pour tout accès ISPE (pas de comptes partagés)
    - [ ] MFA requis pour tous les systèmes ISPE
    - [ ] Délai de session automatique ≤ 15 minutes configuré
    - [ ] Procédure d'accès d'urgence (rupture de verre) documentée et testée
    - [ ] Revues d'accès complétées trimestriellement
    - [ ] Accès des employés licenciés retiré dans les 24 heures
  </Accordion>
  <Accordion title="Chiffrement">
    - [ ] Toutes les ISPE chiffrées au repos (AES-256)
    - [ ] Toutes les ISPE chiffrées en transit (TLS 1.2+)
    - [ ] Les clés de chiffrement gérées par un HSM validé FIPS 140-2+
    - [ ] Calendrier de rotation des clés documenté
    - [ ] Sauvegarde chiffrée des ISPE testée mensuellement
  </Accordion>
  <Accordion title="Contrôles d'audit">
    - [ ] Journalisation d'audit activée pour tout accès ISPE (lecture, écriture, suppression)
    - [ ] Journaux d'audit conservés pendant un minimum de 6 ans
    - [ ] Protection de l'intégrité des journaux d'audit activée (signature HSM)
    - [ ] Journaux d'audit examinés régulièrement (minimum hebdomadaire)
    - [ ] Alertes automatisées sur les modèles d'accès suspects
  </Accordion>
  <Accordion title="Réponse aux violations">
    - [ ] Alertes VaultMonitor configurées pour les indicateurs de violation ISPE
    - [ ] Guide de gestion des violations de 60 jours documenté (Règle de notification de violation HIPAA)
    - [ ] Procédure de transfert des journaux forensiques testée
    - [ ] Équipe de réponse aux incidents formée sur la réponse aux violations HIPAA
  </Accordion>
</AccordionGroup>
