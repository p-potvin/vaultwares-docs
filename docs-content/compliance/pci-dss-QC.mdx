## Aperçu

PCI DSS v4.0 (en vigueur en mars 2024) établit des exigences pour la protection des données de titulaires de carte (DTC). Les organisations qui stockent, traitent ou transmettent des données de titulaires de carte doivent se conformer. Les produits VaultWares répondent directement à plusieurs des 12 exigences de la PCI DSS.

<Note>
  La conformité à la PCI DSS nécessite une validation par un Évaluateur de Sécurité Qualifié (QSA). Ce guide fait le lien entre les capacités de VaultWares et les exigences de la PCI DSS; il ne constitue pas une évaluation de la PCI DSS.
</Note>

## Cartographie des exigences PCI DSS

<Tabs>
  <Tab title="Exigence 3 — Protéger les données stockées">
    **Exigence 3.5**: Les numéros de compte principaux (PAN) doivent être sécurisés partout où ils sont stockés.

    | Sous-exigence | Contrôle VaultWares |
    |---|---|
    | 3.5.1: PAN rendu illisible | Chiffrement AES-256 de VaultCrypt pour les volumes de stockage de DTC |
    | 3.5.1.2: Chiffrement de disque/partition avec des clés non dans le même AD | VaultHSM stocke les clés séparément des données chiffrées |
    | 3.6: Procédures de gestion des clés | Gestion du cycle de vie des clés VaultHSM FIPS 140-3 |
    | 3.7.1: Génération de clés avec des méthodes approuvées | Génération de clés basée sur TRNG de VaultHSM (FIPS 186-5) |
    | 3.7.2: Distribution sécurisée des clés | Enveloppement de clés chiffré; jamais de transport en texte clair |
    | 3.7.4: Changement des clés à la fin du cryptopériode | Rotation automatique des clés VaultHSM avec des cryptopériodes configurables |
    | 3.7.6: Connaissance M-sur-N pour les clés divisées | Gardiennage de clés M-sur-N de VaultHSM |
    | 3.7.7: Prévention de substitution de clés non autorisée | Détection de falsification matérielle; journaux d'audit d'accès aux clés |
    | 3.7.9: Documentation des politiques de gestion des clés | VaultHSM génère des pistes d'audit cryptographiques |

    ```bash
    # Configure CHD volume encryption
    vaultcrypt policy set \
      --name PCI-CHD-Encryption \
      --algorithm AES-256-XTS \
      --key-source hsm \
      --key-label "CHD-Encryption-Key" \
      --apply-to-volumes /data/cardholder \
      --audit-all-access true \
      --require-mfa true

    # Generate a dedicated CHD encryption key
    vw-hsm generate-key \
      --type aes \
      --size 256 \
      --label "CHD-Encryption-Key" \
      --extractable false \
      --usage encrypt,decrypt \
      --cryptoperiod-days 365
    ```
  </Tab>
  <Tab title="Exigence 4 — Protéger en transit">
    **Exigence 4.2**: Une cryptographie forte doit être utilisée pour protéger le PAN lors de la transmission sur des réseaux ouverts et publics.

    ```bash
    # Enforce TLS 1.2+ for all CHD transmission via VaultGate
    vaultgate policy create \
      --name PCI-CHD-Transmission \
      --min-tls-version TLSv1.2 \
      --deny-weak-ciphers true \
      --deny-ssl-v3 true \
      --deny-tls10 true \
      --deny-tls11 true \
      --require-valid-certificates true \
      --apply-to-networks "cardholder-data-environment" \
      --alert-on-violation true
    ```

    Suites de chiffrement approuvées pour la PCI DSS :

    | Suite de chiffrement | Échange de clés | Statut PCI DSS |
    |---|---|---|
    | `TLS_AES_256_GCM_SHA384` | X25519 | ✓ Approuvé |
    | `TLS_AES_128_GCM_SHA256` | X25519 | ✓ Approuvé |
    | `ECDHE-ECDSA-AES256-GCM-SHA384` | ECDHE | ✓ Approuvé |
    | `ECDHE-RSA-AES256-GCM-SHA384` | ECDHE | ✓ Approuvé |
    | `TLS_RSA_WITH_AES_256_CBC_SHA256` | RSA | ⚠ À éviter (pas de PFS) |
    | Tout chiffrement RC4 | — | ✗ Interdit |
    | SSL 3.0 / TLS 1.0 / TLS 1.1 | — | ✗ Interdit |
  </Tab>
  <Tab title="Exigence 7 — Restreindre l'accès">
    **Exigence 7**: Restreindre l'accès aux composants du système et aux données de titulaires de carte selon le besoin de savoir.

    ```bash
    # Create CHD-scoped role with minimal permissions
    vw-admin role create \
      --name CHD-Reader \
      --permissions "storage:chd:read" \
      --scope "environment:cardholder-data" \
      --require-mfa true \
      --require-justification true

    # Apply ABAC policy: CHD access only during business hours from approved IPs
    cat <<'EOF' > /etc/vaultaccess/policies/chd-access.json
    {
      "policy": {
        "name": "CHD-Access-Policy",
        "effect": "allow",
        "principal": {"role": ["CHD-Reader", "CHD-Admin"]},
        "action": ["storage:chd:read", "storage:chd:write"],
        "resource": "arn:vaultwares:storage:*:*:chd-*",
        "condition": {
          "IpAddress": {"request:source_ip": ["10.50.0.0/16"]},
          "StringEquals": {"request:mfa_authenticated": "true"},
          "DateGreaterThan": {"request:utc_time": "08:00:00"},
          "DateLessThan": {"request:utc_time": "20:00:00"}
        }
      }
    }
    EOF

    vw-admin policy apply --file /etc/vaultaccess/policies/chd-access.json
    ```
  </Tab>
  <Tab title="Exigence 8 — Identification des utilisateurs">
    **Exigence 8.3**: Tout accès utilisateur aux composants du système doit être authentifié à l'aide d'au moins un facteur; l'accès privilégié nécessite une MFA.

    ```bash
    # Enforce MFA for all users in the CHD environment
    vw-admin mfa-policy set \
      --name PCI-MFA-Policy \
      --require-mfa true \
      --allowed-methods totp,fido2,push \
      --apply-to-group CHD-Environment-Users

    # Enforce hardware key for privileged access
    vw-admin mfa-policy set \
      --name PCI-Admin-MFA-Policy \
      --require-mfa true \
      --require-hardware-key true \
      --apply-to-roles CHD-Admin,SecurityOfficer
    ```

    **Exigence 8.6.1**: Les comptes partagés/génériques sont interdits.

    ```bash
    # Audit report: detect any shared or generic accounts
    vw-admin audit report shared-accounts \
      --check-last-used true \
      --flag-generic-names true \
      --output /reports/pci-shared-accounts-$(date +%Y%m%d).csv
    ```
  </Tab>
  <Tab title="Exigence 10 — Journaux d'audit">
    **Exigence 10.2**: Les journaux d'audit doivent capturer des événements spécifiques pour tous les composants du système dans l'environnement DTC.

    ```bash
    # Configure PCI-compliant audit logging
    vw-admin audit configure \
      --scope chd-environment \
      --capture-events "user-access,admin-actions,audit-log-access,invalid-access,use-of-identification,init-stop-restart-audit,system-component-creation-deletion,key-management" \
      --retention-days 365 \
      --immediate-export-to-siem true \
      --sign-with hsm:AuditLog-Signing-Key

    # Generate PCI DSS Requirement 10 compliance report
    vw-monitor report generate \
      --template pci-req10-audit \
      --start "2024-10-01" \
      --end "2024-10-31" \
      --format pdf \
      --output /reports/pci-req10-oct-2024.pdf
    ```

    **Exigence 10.5.1**: Les journaux d'audit doivent être protégés contre la destruction et les modifications non autorisées.

    VaultMonitor utilise des journaux d'audit signés par HSM et un stockage de journaux immuable pour prévenir toute altération. Les entrées de journal comprennent une chaîne cryptographique qui peut être vérifiée de manière indépendante.
  </Tab>
</Tabs>

## Segmentation de l'environnement des données de titulaires de carte (CDE)

```bash
# Configure VaultGate to enforce CDE network segmentation
vaultgate segment create \
  --name CDE \
  --networks "10.50.0.0/16" \
  --isolation-level strict

# Allow only specific services from outside the CDE
vaultgate rule create \
  --name Allow-Payment-API \
  --from internet \
  --to CDE:10.50.1.10:443 \
  --protocol tcp \
  --tls-inspection true \
  --log-all true

# Block all other inbound traffic to CDE
vaultgate rule create \
  --name Deny-All-CDE-Inbound \
  --from any \
  --to CDE \
  --action deny \
  --log true
```

## Liste de vérification de conformité PCI DSS

<AccordionGroup>
  <Accordion title="Exigences 3 et 4 — Protection des données">
    - [ ] Tous les PAN stockés chiffrés avec AES-256 (VaultCrypt)
    - [ ] Clés de chiffrement dans un HSM FIPS 140-2+ (VaultHSM)
    - [ ] Le calendrier de rotation des clés correspond aux exigences de cryptopériode PCI
    - [ ] Gardiennage de clés M-sur-N documenté et en place
    - [ ] TLS 1.2+ appliqué à toute transmission de DTC
    - [ ] Suites de chiffrement faibles désactivées (RC4, SSL 3.0, TLS 1.0/1.1)
  </Accordion>
  <Accordion title="Exigences 7 et 8 — Contrôle d'accès">
    - [ ] RBAC de moindre privilège appliqué aux systèmes DTC (VaultAccess)
    - [ ] MFA requis pour tout accès au système DTC
    - [ ] Clé matérielle requise pour l'accès privilégié/administrateur
    - [ ] Aucun compte partagé ou générique dans le CDE
    - [ ] Revues d'accès complétées trimestriellement
    - [ ] Identifiants utilisateur uniques pour tous les comptes
  </Accordion>
  <Accordion title="Exigence 10 — Journalisation des audits">
    - [ ] Journalisation des audits activée sur tous les composants système CDE
    - [ ] Tous les types d'événements requis capturés (Req. 10.2)
    - [ ] Journaux d'audit conservés pendant 12 mois (3 mois en ligne)
    - [ ] Protection de l'intégrité des journaux (signature HSM)
    - [ ] Journaux transférés au SIEM dans les 5 minutes
    - [ ] Procédure de révision quotidienne des journaux en place
  </Accordion>
  <Accordion title="Exigence 12 — Politiques de sécurité">
    - [ ] Politique de sécurité mise à jour pour faire référence aux contrôles VaultWares
    - [ ] Plan de réponse aux incidents incluant les procédures forensiques VaultWares
    - [ ] Politique de gestion des clés documentée selon l'Exig. 3.7.9
    - [ ] Formation annuelle de sensibilisation à la sécurité incluant les pratiques de chiffrement
  </Accordion>
</AccordionGroup>
