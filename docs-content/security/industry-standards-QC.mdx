## NIST SP 800-57 — Key Management

VaultHSM implements the key management lifecycle defined in NIST SP 800-57 Parts 1–3.

| Lifecycle phase | VaultHSM capability |
|---|---|
| Key generation | TRNG-based key generation; FIPS 186-5 compliant |
| Key establishment | NIST SP 800-56A/B ECDH and RSA key agreement |
| Key storage | Tamper-resistant secure element; keys never exported in plaintext |
| Key distribution | Encrypted key wrapping (AES-256 or RSA-4096 OAEP) |
| Key use | Enforced algorithm-key size pairings; use count tracking |
| Key backup | M-of-N key splitting; encrypted backup with custodian keys |
| Key archival | Archive keys with retention metadata; audit log of all access |
| Key revocation | Immediate zeroization on command; CRL publication for certificates |
| Key destruction | NIST-compliant cryptographic erasure; physical destruction on demand |

## Zero Trust Architecture (NIST SP 800-207)

VaultAccess implements all seven tenets of the Zero Trust Architecture:

<AccordionGroup>
  <Accordion title="1. All data sources and computing services are resources">
    VaultAccess treats all devices — including VaultDrive, VaultHSM, and VaultScan — as resources that must be authenticated and authorized for each access request.
  </Accordion>
  <Accordion title="2. All communication is secured regardless of network location">
    All communications are encrypted with TLS 1.3. VaultGate enforces mutual TLS (mTLS) for server-to-server communication within the enterprise perimeter.
  </Accordion>
  <Accordion title="3. Access to individual enterprise resources is granted per-session">
    VaultAccess issues time-limited access tokens scoped to the specific resource and operation. Standing access is minimized through PAM just-in-time provisioning.
  </Accordion>
  <Accordion title="4. Access to resources is determined by dynamic policy">
    ABAC policies evaluate user identity, device posture, location, time, and risk score at every access request using the VaultAccess policy engine.
  </Accordion>
  <Accordion title="5. Monitor and measure integrity and security posture of assets">
    VaultMonitor continuously collects device telemetry and security events, feeding the policy engine's risk score calculations in real-time.
  </Accordion>
  <Accordion title="6. Authentication and authorization are dynamic and strictly enforced">
    Adaptive MFA escalates authentication requirements based on risk signals. VaultAccess can require step-up authentication mid-session when anomalies are detected.
  </Accordion>
  <Accordion title="7. Collect as much information as possible to improve posture">
    All access decisions, device events, and authentication events are logged to the VaultMonitor audit store and forwarded to SIEM. Data feeds back into policy refinement.
  </Accordion>
</AccordionGroup>
