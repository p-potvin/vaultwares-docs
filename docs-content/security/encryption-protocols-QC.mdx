## Vue d'ensemble

Les produits VaultWares utilisent uniquement des algorithmes cryptographiques standards et bien révisés. Ce document offre une plongée technique approfondie dans les protocoles de chiffrement, les fonctions de dérivation de clés et les configurations de protocole utilisées dans toute la famille de produits VaultWares.

## Chiffrement symétrique

### AES-256-XTS (chiffrement de stockage)

Tous les produits de stockage VaultWares (VaultDrive, VaultCrypt) utilisent AES-256-XTS pour le chiffrement des disques et des volumes.

**Pourquoi XTS?**
XTS (XEX-based Tweaked CodeBook mode with ciphertext Stealing) est la norme IEEE 1619-2007 pour le chiffrement de stockage. Contrairement aux modes ECB ou CBC, XTS élimine les motifs dans le texte chiffré même lorsque les blocs de texte clair sont similaires, car chaque secteur de 512 octets est chiffré avec un *tweak* unique dérivé du numéro de secteur.

```
XTS-AES-256 key structure:
  Key = Key₁ (256-bit) ‖ Key₂ (256-bit) = 512 bits total
  Key₁: used to encrypt the sector data
  Key₂: used to encrypt the sector number (the tweak)
  Ciphertext[i] = AES_K₁(Plaintext[i] ⊕ T[i]) ⊕ T[i]
  T[i] = AES_K₂(SectorNumber) · α^i  (in GF(2^128))
```

| Paramètre | Valeur |
|-----------|-------|
| Taille de clé | 256 bits (chaque sous-clé) |
| Taille de bloc | 128 bits |
| Taille de secteur | 512 octets (matériel), configurable en logiciel |
| Standard | IEEE 1619-2007, NIST SP 800-38E |

### AES-256-GCM (données en transit)

Toutes les connexions TLS et les communications API utilisent AES-256-GCM, qui fournit à la fois la confidentialité et l'authentification (mode AEAD).

```python
# Example: AES-256-GCM encryption using VaultWares Python SDK
from vaultwares import VaultWaresClient

client = VaultWaresClient(api_key="vw_live_...")

# Encrypt data with a key managed by VaultHSM
result = client.crypto.encrypt(
    key_label="App-Data-Key",
    algorithm="AES-256-GCM",
    plaintext=b"sensitive data",
    aad=b"application-context-string"  # Additional Authenticated Data
)

ciphertext = result.ciphertext  # bytes
iv = result.iv                  # 96-bit nonce
auth_tag = result.auth_tag      # 128-bit authentication tag

# Decrypt
plaintext = client.crypto.decrypt(
    key_label="App-Data-Key",
    algorithm="AES-256-GCM",
    ciphertext=ciphertext,
    iv=iv,
    auth_tag=auth_tag,
    aad=b"application-context-string"
)
```

## Cryptographie asymétrique

### RSA

RSA est utilisé pour l'encapsulation de clés, les signatures de certificats et la compatibilité héritée.

| Cas d'utilisation | Taille de clé | Remplissage | Standard |
|----------|----------|---------|----------|
| Certificats serveur TLS | 2048–4096 bits | PKCS#1 v2.1 OAEP / PSS | NIST SP 800-56B |
| Enveloppement de clés (DEKs) | 4096 bits recommandé | OAEP avec SHA-256 | NIST SP 800-56B |
| Signature de code | 4096 bits | PSS avec SHA-384 | FIPS 186-5 |

<Warning>
  VaultWares ne prend plus en charge RSA-1024. Toutes les nouvelles générations de clés par défaut sur RSA-4096. Les clés RSA-2048 existantes sont acceptées, mais signalées pour rotation dans la console de gestion des clés.
</Warning>

### Cryptographie à courbe elliptique (ECC)

ECC fournit une sécurité équivalente à RSA avec des tailles de clés beaucoup plus petites, ce qui réduit la surcharge de calcul — important pour le débit des HSM.

| Courbe | Taille de clé | RSA équivalent | Utilisation principale |
|-------|----------|---------------|-------------|
| P-256 (secp256r1) | 256 bits | RSA-3072 | TLS, signature ECDSA |
| P-384 (secp384r1) | 384 bits | RSA-7680 | Signature haute sécurité, Suite B |
| P-521 (secp521r1) | 521 bits | RSA-15360 | Matériau de clé à long terme |
| X25519 | 255 bits | — | Échange de clés ECDH (TLS 1.3) |
| Ed25519 | 255 bits | — | Signatures numériques rapides |

```bash
# Generate an ECC key pair on VaultHSM
vw-hsm generate-key \
  --type ecc \
  --curve P-384 \
  --label "Signing-Key-ECDSA-P384" \
  --usage sign,verify \
  --extractable false

# Sign a document
vw-hsm sign \
  --key "Signing-Key-ECDSA-P384" \
  --algorithm ECDSA-SHA384 \
  --input document.pdf \
  --output document.sig
```

## Configuration TLS

### TLS 1.3 (préféré)

Tous les composants serveur VaultWares préfèrent TLS 1.3. TLS 1.2 est pris en charge pour la compatibilité avec les clients hérités, mais il est configurable.

**Suites de chiffrement TLS 1.3 supportées par VaultWares :**

| Suite de chiffrement | Échange de clés | Authentification | Chiffrement | Hachage |
|---|---|---|---|---|
| `TLS_AES_256_GCM_SHA384` | X25519/P-256 | — | AES-256-GCM | SHA-384 |
| `TLS_CHACHA20_POLY1305_SHA256` | X25519 | — | ChaCha20-Poly1305 | SHA-256 |
| `TLS_AES_128_GCM_SHA256` | X25519/P-256 | — | AES-128-GCM | SHA-256 |

**Suites de chiffrement TLS 1.2 (lorsqu'requis pour la compatibilité) :**

| Suite de chiffrement |
|---|
| `ECDHE-ECDSA-AES256-GCM-SHA384` |
| `ECDHE-RSA-AES256-GCM-SHA384` |
| `ECDHE-ECDSA-CHACHA20-POLY1305` |

```bash
# Configure TLS settings
vw-admin tls configure \
  --min-version TLSv1.2 \
  --prefer-TLS13 true \
  --ciphers "TLS_AES_256_GCM_SHA384:TLS_CHACHA20_POLY1305_SHA256:ECDHE-ECDSA-AES256-GCM-SHA384" \
  --session-tickets false \
  --renegotiation disabled \
  --hsts-max-age 31536000
```

## Fonctions de dérivation de clés

### PBKDF2 (dérivation de clés basée sur un mot de passe)

VaultDrive utilise PBKDF2 pour dériver la clé de chiffrement du volume à partir du mot de passe de l'utilisateur.

```
PBKDF2-HMAC-SHA512:
  DK = PBKDF2(PRF, Password, Salt, c, dkLen)
  PRF: HMAC-SHA512
  Salt: 256-bit random, stored in the device's secure element
  c: ≥ 600,000 iterations (NIST SP 800-132 recommendation for SHA-512)
  dkLen: 512 bits (used as the XTS key pair for AES-256-XTS)
```

<Note>
  Le compte d'itérations sur le matériel VaultDrive est lié à un minuteur matériel plutôt qu'à une boucle logicielle, ce qui le rend constant dans le temps quelle que soit la vitesse du processeur hôte. Cela empêche le compte d'itérations d'être contourné trivialement sur des machines haute performance.
</Note>

### HKDF (dérivation de clés basée sur HMAC)

Utilisé pour dériver les clés de session et le matériel de clés à partir des clés maîtresses.

```
HKDF (RFC 5869):
  HKDF-Extract: PRK = HMAC-Hash(Salt, IKM)
  HKDF-Expand: OKM = T(1) ‖ T(2) ‖ ... ‖ T(⌈L/HashLen⌉)
  Hash: SHA-256 or SHA-384
```

```python
# Using HKDF for session key derivation (VaultWares SDK)
from vaultwares.crypto import derive_key

session_key = derive_key(
    algorithm="HKDF-SHA256",
    master_key_label="App-Master-Key",
    info=b"session-2024-11-15T14:00:00Z",
    length=32  # 256 bits
)
```

### X9.63 KDF (accord de clés HSM)

Utilisé dans l'accord de clés ECIES (Elliptic Curve Integrated Encryption Scheme) au sein de VaultHSM.

## Algorithmes de hachage

| Algorithme | Taille de sortie | Standard | Utilisation chez VaultWares |
|-----------|-------------|----------|-------------------|
| SHA-256 | 256 bits | FIPS 180-4 | HMAC, TLS, intégrité générale |
| SHA-384 | 384 bits | FIPS 180-4 | Signatures de certificats (clés P-384) |
| SHA-512 | 512 bits | FIPS 180-4 | PBKDF2, HMAC haute sécurité |
| SHA3-256 | 256 bits | FIPS 202 | Disponible dans VaultHSM Enterprise |
| BLAKE3 | 256 bits | — | Sommes de contrôle haute performance dans VaultBackup |

<Warning>
  MD5 et SHA-1 ne sont pas pris en charge dans aucun produit VaultWares pour des raisons de sécurité. Les fichiers de configuration ou les intégrations qui font référence à ces algorithmes seront rejetés.
</Warning>

## Préparation post-quantique

VaultWares se prépare activement à la transition vers la cryptographie post-quantique.

<AccordionGroup>
  <Accordion title="Statut actuel">
    VaultHSM Enterprise (firmware 5.5+) prend en charge les algorithmes de cryptographie post-quantique (PQC) NIST suivants en mode aperçu :
    - **ML-KEM** (CRYSTALS-Kyber) — encapsulation de clés, NIST FIPS 203
    - **ML-DSA** (CRYSTALS-Dilithium) — signatures numériques, NIST FIPS 204
    - **SLH-DSA** (SPHINCS+) — signatures basées sur hachage, NIST FIPS 205
  </Accordion>
  <Accordion title="Échange de clés hybride">
    Pour les infrastructures critiques, VaultWares recommande l'échange de clés hybride : combiner ECDH classique (X25519) avec ML-KEM pour se protéger contre les attaques classiques et quantiques pendant la période de transition.

    ```bash
    # Enable hybrid TLS key exchange on VaultGate
    vaultgate tls-policy set \
      --key-exchange "X25519+ML-KEM-768" \
      --experimental-pqc true
    ```
  </Accordion>
  <Accordion title="Calendrier de migration">
    VaultWares vise un support PQC complet dans le canal stable d'ici 2026, aligné sur le calendrier de migration recommandé par le NIST. Contactez [security@vaultwares.com](mailto:security@vaultwares.com) pour la planification de migration d'entreprise.
  </Accordion>
</AccordionGroup>
