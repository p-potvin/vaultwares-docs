<Note>
**Source de vérité de niveau 1.** Ce document est normatif pour tous les projets VaultWares.
Un projet qui ne peut pas satisfaire ces exigences DOIT ajouter `NOTE.encryption.md`
à la racine de son dépôt pour expliquer la raison technique et le mécanisme de repli choisi.
</Note>

## Portée

Cette norme régit chaque endroit où un projet VaultWares chiffre les données utilisateur
côté client à l'aide de secrets contrôlés par l'utilisateur — les coffres-forts locaux, les sauvegardes chiffrées,
les charges utiles de synchronisation appareil-à-appareil, la messagerie de bout en bout, et tout flux où le
serveur ne doit pas être en mesure de lire le contenu utilisateur.

Elle ne régit **pas** le chiffrement de transport (TLS) ou le chiffrement de stockage effectué
par les produits matériels VaultWares sous garde de clés d'entreprise — ceux-ci sont couverts
par [Protocoles de chiffrement](./encryption-protocols).

## Architecture obligatoire

Le chiffrement côté client DOIT utiliser **ML-KEM (FIPS 203)** comme mécanisme d'encapsulation de clé, superposé à **AES-256-GCM** comme chiffre de données symétrique, selon le modèle de chiffrement par enveloppe ci-dessous.

```
PIN / passphrase
   │
   ▼  Argon2id (m = 64 MiB, t = 3, p = 1, salt = 16 random bytes)
master KEK (AES-256)
   │
   ├──► wraps the long-term ML-KEM private key (AES-256-GCM)
   │
   │    long-term ML-KEM public key  ── stored plaintext, exportable
   │    long-term ML-KEM private key  ── only ever in memory after PIN entry
   │
   ▼
For each blob (preview, message, backup chunk):
   1. Generate random 32-byte DEK (crypto.getRandomValues)
   2. Encrypt blob with AES-256-GCM under DEK; store ciphertext + IV + tag
   3. ML-KEM encapsulate against public key → (ML-KEM ciphertext, shared secret K_s)
   4. AES-256-Key-Wrap (RFC 5649) the DEK under K_s
   5. Persist: ML-KEM ciphertext ‖ wrapped DEK ‖ blob ciphertext
```

## Sélection des paramètres

| Niveau | Ensemble de paramètres | Quand l'utiliser |
|------|---------------|-------------|
| **Par défaut** | **ML-KEM-1024** | Toutes les données confidentielles à long terme. Niveau de sécurité NIST 5. |
| **Minimum autorisé** | **ML-KEM-512** | Seulement lorsque la latence de décapsulation dégrade directement la performance perçue par l'utilisateur (ex. décodage de prévisualisation en temps réel sur des chemins chauds). Niveau de sécurité NIST 1. Documentez le budget de latence qui justifie ce choix dans `NOTE.encryption.md`. |
| **Repli (avec note)** | **AES-256-GCM seulement**, pas de couche PQ | Seulement lorsque ML-KEM ne peut pas être implémenté dans l'environnement d'exécution du projet (ex. contraintes SDK strictes, pas de bibliothèque PQ auditée disponible). Nécessite `NOTE.encryption.md` à la racine du dépôt. |

ML-KEM-768 n'est pas intentionnellement listé — les projets ont soit besoin d'une marge de grade quantique
(utiliser 1024) soit sont limités par la latence (utiliser 512). Choisir le milieu est rarement la bonne
réponse; si vous pensez que c'est le cas, écrivez le compromis dans `NOTE.encryption.md` et
escaladez.

## Propriétés requises

Une implémentation conforme DOIT garantir tout ce qui suit.

### Connaissance zéro

- Le serveur NE DOIT PAS recevoir le code PIN, la KEK maître, la clé privée, la DEK,
  ou les blocs en texte clair.
- Les sauvegardes téléchargées pour la synchronisation inter-appareils DOIVENT être chiffrées avec la clé publique
  avant de quitter l'appareil.
- Les télémétries, journaux et rapports de plantage NE DOIVENT PAS contenir de matériel de clé ou de texte clair.

### Gestion des clés

- Les codes PIN ne DOIVENT JAMAIS être stockés sous une forme réversible. Stocker uniquement le hachage dérivé d'Argon2id
  pour la vérification, avec un sel aléatoire par utilisateur.
- Le sel Argon2id DOIT être stocké avec le hachage de vérification.
- Les clés symétriques NE DOIVENT PAS être dérivées directement d'un code PIN sans une KDF lente.
- L'utilisation directe d'un code PIN comme clé AES est interdite.
- Les clés privées DOIVENT être enveloppées sous la KEK maître chaque fois qu'elles sont persistées.
- Les DEK DOIVENT être éphémères (régénérés par bloc) — ne jamais réutiliser une DEK entre les blocs.

### Agilité cryptographique

- Le format sur disque DOIT inclure un identifiant d'algorithme et un champ de version afin que l'ensemble de paramètres puisse changer dans une future version sans casser les coffres-forts existants.
- Un chemin de migration documenté DOIT exister pour mettre à niveau les anciens coffres-forts vers l'ensemble de paramètres actuel.

## Guide d'implémentation

### Bibliothèques recommandées

| Environnement d'exécution | Bibliothèque | Notes |
|---------|---------|-------|
| Navigateur / Node (TypeScript) | [`@noble/post-quantum`](https://github.com/paulmillr/noble-post-quantum) | Audité, zéro dépendance, MIT |
| Navigateur / Node (TypeScript) | [`liboqs-js`](https://github.com/open-quantum-safe/liboqs) | Algorithmes de référence NIST, charge utile plus grande |
| Natif (Rust) | [`pqcrypto`](https://github.com/rustpq/pqcrypto) | Enveloppe des implémentations de référence PQClean |
| Python | [`pyoqs`](https://github.com/open-quantum-safe/liboqs-python) | Liens OQS |

Choisissez une bibliothèque qui possède un audit de sécurité tiers sur l'implémentation ML-KEM. Si aucune n'est disponible pour votre environnement d'exécution, documentez l'écart dans `NOTE.encryption.md` et préférez le repli AES-256-GCM à une implémentation PQ non auditée.

### Modèle `NOTE.encryption.md`

Lorsqu'un repli ou un écart par rapport à cette norme est inévitable, ajoutez le
fichier suivant à la racine du dépôt :

```markdown
# Encryption Note

**Standard reference:** vaultwares-docs/security/zero-knowledge-encryption-standard

**Deviation:** <ML-KEM-512 instead of -1024 / AES-only fallback / etc.>

**Reason:** <one paragraph: technical constraint, latency budget, library gap>

**Mitigation:** <what we do instead, what we monitor, when we plan to revisit>

**Owner:** <name / role>
**Reviewed:** <YYYY-MM-DD>
```

## Schémas interdits

Les schémas suivants ne sont pas conformes et DOIVENT être supprimés lorsqu'ils sont découverts :

- Le stockage du code PIN, du mot de passe ou de la phrase de récupération de l'utilisateur en texte clair ou sous toute forme réversible
- L'utilisation d'un code PIN, d'un mot de passe ou d'un secret de faible entropie directement comme clé AES
- L'utilisation d'un hachage rapide (SHA-1, SHA-256 en un seul passage, MD5) comme hachage de mot de passe
- La réutilisation d'une DEK entre plusieurs blocs
- L'envoi de données utilisateur non chiffrées à n'importe quel serveur, même temporairement, même « pour diagnostic »
- Les clés de chiffrement codées en dur dans le code source, les artefacts de construction ou les fichiers d'environnement
- Les données « chiffrées » dont la clé est expédiée avec elles (le soi-disant théâtre du chiffrement)

## Audit et révision

- Chaque projet soumis à cette norme DOIT être révisé annuellement par rapport à la version actuelle de ce document. La date de révision et le réviseur doivent être inscrits dans `NOTE.encryption.md` même si le projet est entièrement conforme.
- Les nouvelles implémentations DOIVENT faire réviser leur flux cryptographique par au moins un ingénieur qui n'a pas écrit l'implémentation.
- Le format de transmission (wire format) et le format sur disque DOIVENT être documentés dans le propre
  README ou fichier de notes de sécurité du projet afin que le format puisse être analysé par des outils de récupération
  à l'avenir.
