## Context

`fullxxx.video` and `prom-king.xyz` are both WordPress installs driven by two custom plugins (`fullxxx-video`, `promking-tube`) and a `tube-shared` helper plugin. The catalog is small (≈ 270 active videos, 132 studios, 245 actors), the UX is read-heavy with a few islands of interactivity (player modal, search, consent banner, age gate, admin), and **none of WordPress's headline features are in use**: no built-in theme, no posts/pages editor for content, no comments, no media library, no plugin ecosystem, no Gutenberg, no XML-RPC, no users.

What the WP install actually buys us today:

- A relational store with a custom-post-type + taxonomy + term-meta abstraction
- A cron loop
- An admin UI we extended with bolted-on bulk actions and a Gender field
- A rewrite-rules subsystem we keep fighting (the `/2257/ → year=2257` collision being the most recent)
- A search hook stack
- A sitemap generator we partially overrode

Every single one of these is replaceable in a few hundred lines of code.

What the WP install costs us:

- Two PHP versions on the box (8.2 for fpm, 8.4 for wp-cli) with extension mismatches that silently kill features (`mysqli`, `curl`, `xml` all bit us in the last 48 hours).
- Deploy scripts that have to dance around `www-data` ownership, `vwdeploy` permissions, `flock` files, and wp-cli's "YIKES, don't run as root."
- Dev velocity tax: PHP/WP idioms (hooks, filters, template loaders, `WP_Query` arg surfaces, `rewrite_rules_array`) slow iteration when the operator is fluent in React/TypeScript.
- Admin UX ceiling: every operator workflow we added (`fxv_actor_gender_*`, bulk reclassify, fetcher trigger) is grafted onto `wp-admin` and reads as exactly that — grafted on.
- A second mental model for monitoring/logging — WP's own log surface vs. the rest of the VaultWares fleet (Health Ledger, Alarm Joker, Token Joker).

The owner has flagged two specific pain drivers: **dev velocity** and **operator UX**. Both point at the same conclusion: the WP overhead is paying for nothing we use, and the rebuild surface is small enough to be done in one stint.

## Decision

We **retire the WordPress installs at `fullxxx.video` and `prom-king.xyz`** and replace them with a four-piece architecture:

1. **`Prom-King/shared-tube`** — new repo, pnpm monorepo.
2. **`shared/`** — npm package: design tokens, Astro components (header, footer, card, player, consent banner, age gate, ad-slot wrapper), fetcher engine (per-source XPath configs + normalize), Drizzle schema, shared TypeScript types.
3. **`apps/fxv` + `apps/pkt`** — thin Astro apps. Hybrid SSR: legal pages and the home shell are pre-rendered at build time (SSG, served by nginx); catalog pages (listings, single video, taxonomy archives, search, admin) are SSR per request. Each app composes `shared/` with a per-site config (brand tokens, source list, GTM container, ad codes, legal text).
4. **`vaultwares-api`** — extended with a new **Prom-King router** that owns: video/actor/studio/category CRUD, fetcher cron + manual trigger, settings, stats, auth integration (deferred enable). DB writes live here. Logging/monitoring stays uniform with the rest of the fleet.

Postgres replaces MySQL. Drizzle replaces the WP query surface. React replaces the entire `wp-admin` extension layer for our workflows. Astro replaces the template loader + rewrite rule layer.

**Migration is a hard cutover with a fresh catalog**: we drop the WP DB and refetch from the upstream sources after the new stack ships. The current catalog is small (≈ 270 videos) and refetchable, so the data-migration risk is preferable to one-shot ETL out of `wp_posts` + `wp_postmeta` + `wp_terms`.

**Branch strategy**: all work happens on a feature branch in the new repo; the existing `Prom-King/tube-sites` WP install stays running until the new stack passes a local smoke test, then they replace the old ones.

## Options Considered

### Option A — Keep WordPress, fix what's broken in place (do nothing)

| Dimension          | Assessment                                                   |
| ------------------ | ------------------------------------------------------------ |
| Complexity         | Low (no rewrite)                                             |
| Cost               | Low up front, accumulating tax                               |
| Scalability        | Adequate for ≈ 1000 videos per site                          |
| Team familiarity   | Already there                                                |
| Maintenance burden | High — WP + PHP version drift, deploy script fragility, rewrite-rule edge cases |

**Pros:** No rewrite. Catalog stays. Nothing to learn.

**Cons:** Operator UX stays grafted-on. Dev velocity stays slow. Every new feature negotiates with WP's mental model. Monitoring stays bifurcated.

### Option B — Headless CMS (Directus or Payload on Postgres) + Astro frontends

| Dimension          | Assessment                               |
| ------------------ | ---------------------------------------- |
| Complexity         | Medium (two new components: CMS + Astro) |
| Cost               | Low for CMS (free OSS), medium for setup |
| Scalability        | Excellent                                |
| Team familiarity   | New (CMS-specific quirks)                |
| Maintenance burden | Medium (own + upgrade the CMS)           |

**Pros:** Free admin UI on day one. Auto-generated REST/GraphQL. Clean schema separation.

**Cons:** Owner explicitly disliked CMS UX — "they're made for non-technical people, it makes it harder for me rather than simply coding what i want." **(Addendum: you can't quote someone if you're gonna rephrase in your own words bruh)** The page system, theming, shortcodes friction of WP would be replaced by similar friction in Directus. Doesn't address the operator-UX pain driver.

**Rejected** on owner preference + the fact that the admin we need (bulk taxonomy edit, one-button fetcher, stats, canned SQL) is custom enough that a generic CMS would be in the way more than it would help.

### Option C — Astro + Node API + Postgres, per-site backend (no central pipelines)

| Dimension          | Assessment                       |
| ------------------ | -------------------------------- |
| Complexity         | Medium                           |
| Cost               | Low                              |
| Scalability        | Good                             |
| Team familiarity   | High (React/TS comfort)          |
| Maintenance burden | Medium — two backends to monitor |

**Pros:** Clean per-site boundary. Cheaper for the FXV/PKT scope alone.

**Cons:** Bifurcates monitoring/logging from the rest of the VaultWares fleet. Auth would have to be re-invented per site or punted forever. Misses the synergy with `vaultwares-api` becoming the central API.

**Rejected** in favour of Option D after the owner confirmed `vaultwares-api` is becoming the canonical API for every VaultWares + Prom-King app.

### Option D — Astro frontends + Prom-King router in vaultwares-api + Postgres (chosen)

| Dimension          | Assessment                                                   |
| ------------------ | ------------------------------------------------------------ |
| Complexity         | Medium — one new code surface (Astro) + one extension to existing API |
| Cost               | Low (shares infra with the rest of the fleet)                |
| Scalability        | Excellent                                                    |
| Team familiarity   | High                                                         |
| Maintenance burden | Low — one DB, one log stream, one auth surface ready to flip on |

**Pros:** One API to monitor. One auth implementation. One DB connection pool. One logging story. Reusable design tokens (vault-flows). Owner velocity goes up because the iteration loop matches the rest of their fleet.

**Cons:** `vaultwares-api` now hosts tube-specific business logic alongside everything else. Boundary is enforced by a `routes/promking/` subtree and a `db/promking/` schema. Worth the trade because the central monitoring story is what the owner explicitly named as the reason to do this.

## Trade-off Analysis

The single most consequential trade-off is **Option B vs. Option D**: lean on someone else's admin (free, mediocre fit) or build our own (custom, exactly right). The owner is a technical operator who wants `console.log + button + bulk action + canned SQL`, not Gutenberg-style page composition. That's a 1–2 day build, not a 1–2 week CMS integration, and it pays for itself the first time we want to ship a workflow the CMS doesn't model cleanly.

The second consequential trade-off is **hard cutover vs. strangler fig**. Strangler is the textbook-safe answer for a real migration. Here, the catalog is small enough (≈ 270 videos), the refetch loop is automatic (pornxp + 1porn are already producing 60+ videos per run), and the legacy install is small enough that the cost of running parallel surfaces outweighs the cutover risk. Hard cutover means simpler code in v1 (no compatibility layer with `_fxv_*` postmeta keys) and no orphaned WP routes leaking into the new world.

The third trade-off is **Astro hybrid SSR vs. SSG**. Pure SSG buys the cheapest runtime but forces a build-on-fetch loop that gets fragile when the catalog refreshes hourly. Hybrid SSR gives static delivery for legal/home (which never changes between deploys) and live rendering for catalog (which always changes), without paying SSR cost for the bits that don't need it.

## Consequences

### What becomes easier

- Every admin workflow is React behind `vaultwares-api` auth. Adding a new bulk action is one file under `apps/<site>/src/admin/`.
- Adding a new source to the fetcher is one config block + one normalize function in `shared/src/fetcher/sources/`, no rewrite-rule dance, no detail-page extractor hooks unless the source actually needs them.
- Stats, monitoring, and Health Ledger integration flow through `vaultwares-api` like every other VaultWares app.
- React tokens / vault-flows components compose directly into the catalog.
- Deploys are one `pnpm build` + one rsync per app. No PHP version dependency, no plugin dirs to chown.
- Cutover **deletes ~5000 lines of PHP** that exist because WP needs them, not because the catalog does.

### What becomes harder

- We own the admin UX. No "just install a plugin" escape hatch. (We didn't have one with WP either, but the temptation will be gone.)
- We own the auth surface (deferred enable, but the code path has to be designed in).
- Search migrates from WP's `WP_Query` + custom filters to Postgres FTS. v1 lands the FTS index; tuning is a v1.1 problem.
- Sitemap, JSON-LD, canonical, og/twitter meta all become explicit Astro components instead of WP filter chains. (Clearer, but more code in the diff.)

### What we'll need to revisit

- **Auth turn-on date.** Schema lands in v1 (users table + favourites + roles). UI lands when the owner says go.
- **The PKT WordPress install** at `prom-king.xyz`. Same plan, same monorepo, parallel cutover. v1 ships both apps; the PKT-specific work (4 upstream sources, the wipe-and-rebackfill admin button) carries over via the same `shared/fetcher` engine with different configs.
- **Health Ledger integration** for deploy alerts replaces the old GitHub-issue + `@copilot` pipeline (now disabled per `operations/deploy-alerts.mdx`).

## Implementation deviations from this ADR

These are intentional — anywhere this ADR's prose conflicts with what's in the repo, the repo wins. Listed here so reviewers don't trip on stale phrasing.

1. **`output: 'hybrid'` → default `static` + per-page `prerender = false`.** Astro 5 removed the `hybrid` output mode; `static` is now the default and behaves identically. SSR pages (`videos/`, `videos/[slug]`, taxonomy archives, `search.astro`, `sitemap.xml.ts`, `/admin/*`) opt in with `export const prerender = false;`. Where this ADR says "hybrid SSR", read "static-default + per-page SSR opt-out."
2. **Fetcher CLI entrypoint.** The `shared/` file tree in this ADR doesn't show `shared/src/fetcher/cli.ts`. It exists and is the contract surface for FastAPI → fetcher (NDJSON over stdout). See the in-repo `docs/router-integration.md` for the wire format.
3. **`fullvideos.xxx` selectors are now locked.** The ADR mentioned modeling after this site; the actual listing-page + detail-page selectors landed during scaffold based on owner-provided DOM notes. See `shared/src/fetcher/sources/fullvideos.ts` and `docs/fetcher-sources.md` — frontpage DMCA placeholders are filtered via `skipItemIf`.
4. **`node-cron` → APScheduler.** The router lives on the Python (FastAPI) side of `vaultwares-api`, so the nightly fetch cron uses APScheduler in-process (`app/routers/promking/cron.py`) instead of node-cron. Schedule configuration is per-(site, source), stored in the `settings` table under key `fetcher_cron`, and reloadable without restart. Same intent as the ADR's "node-cron for the scheduled run" — just on the Python runtime.
5. **`pornxp.com` → `pornxp.bz`** (was `.fo` in the PHP-port; `.fo` 301s to `.bz`, we pin `.bz` directly to skip the hop). **`1porn.com` → `1porn.tv`.** Anywhere the ADR or earlier docs say `.com`, the working host is the modern variant.
6. **Apps run from `/srv/repos`, not `/var/www`.** The original cutover plan rsync'd `apps/<site>/dist/` into `/var/www/<domain>/` after `pnpm -r build` and pointed systemd `ExecStart` at that copy. `@astrojs/node` standalone externalises `react`/`react-dom`; without the workspace pnpm symlinks under `shared-tube/node_modules/.pnpm/...`, Node crashes with `ERR_MODULE_NOT_FOUND`. Systemd units now run from `/srv/repos/Prom-King/shared-tube/apps/<site>/dist/server/entry.mjs` with `WorkingDirectory=/srv/repos/Prom-King/shared-tube/apps/<site>/`. `/var/www/<domain>/` is no longer a deploy target. The deploy script no longer rsyncs.
7. **xvideos is 1-indexed live**, contrary to the PHP-port's assumption of 0-indexed. `/new/0` now returns 404; `/new/1` is the first page. `Math.max(0, page-1)` was wrong; the source config now uses `page` directly.
8. **`fullxxx.video` HTTPS via certbot, not pre-existing.** Greencloud-vps had a letsencrypt cert for `prom-king.xyz` but not `fullxxx.video` (the old WP install was broken externally anyway — both domains were already returning 404 before cutover). New cert obtained 2026-06-06 via `certbot certonly --webroot -w /var/www/_acme -d fullxxx.video -d www.fullxxx.video`. Auto-renewal scheduled.
9. **Admin browser → API path is same-origin via nginx**, not direct-to-tailnet. The first admin design had a configurable API-base field defaulting to `http://100.71.101.21:9001`. From an HTTPS page that's blocked by mixed-content + CORS. nginx now proxies `/api/promking/*` and `/auth/*` to pipelines, injecting `x-vw-gateway-secret` from `/etc/nginx/snippets/vw-gateway-secret.conf` (0640 root:www-data; never committed). The admin React code uses `window.location.origin`; the override field was removed after stale localStorage values caused operators to keep hitting cross-origin URLs.
10. **CLI emits one `videos` event per item, not one big array.** A 3-page `fullvideos` run produced a single ~80 KB `videos` line that blew past `asyncio.StreamReader`'s 64 KiB default. FastAPI's `readline()` silently failed and the persistence path never ran. The CLI now emits one `videos` event per item (single-element array each); FastAPI accumulates. As belt-and-braces, the subprocess `StreamReader` `limit` is raised to 10 MiB.
11. **`_persist_videos` is per-row autocommit, no batch transaction.** The first design wrapped the batch in `async with conn.transaction():`. A single `NOT NULL` violation (e.g. `embed_url` empty on a fullvideos card with no `data-preview`) aborted the Postgres transaction; the inner `try/except` swallowed the immediate error but subsequent `INSERT`s all raised "current transaction is aborted, commands ignored", and the outer manager re-raised at commit — so `_finalize_run` never ran. Each row is now its own autocommit `INSERT … ON CONFLICT (site, source_url) DO NOTHING`. FastAPI also emits a `persisted` SSE event after persistence with the real `added` count, since the CLI's `done` always reports `added: 0`.
12. **No public bearer fallback for the admin path.** `vaultwares-api` (FastAPI) gates `/auth/me`, `/auth/login`, `/auth/api-keys` against `BOOTSTRAP_ADMIN_*` in its `.env`. Tokens are minted via `POST /auth/login` (1h JWT) or `POST /auth/api-keys` (long-lived). The admin React app verifies against `/auth/me` (not `/me`) and persists the token in localStorage.
