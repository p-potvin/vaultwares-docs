# Renommage

- ID de catégorie : `RENAMING`
- Source du résumé : `instructions/summaries/RENAMING.md`
- Source des notes : `instructions/notes/RENAMING.md`

S'applique quand : on renomme un dépôt GitHub, un projet Jira, ou tout identifiant suivi auquel d'autres systèmes font référence.
À faire :
- Avant de renommer quoi que ce soit sur GitHub, vérifier `vw-jira-sync/config.yaml` — chaque dépôt a une clé de projet Jira et un fichier de mappage.
- Suivre la procédure complète de 6 étapes pour renommer un dépôt GitHub (voir les notes ou `operations/jira-sync.mdx`) :
  1. Renommer sur GitHub.
  2. Mettre à jour `repo_project_keys` et `repos` dans `vw-jira-sync/config.yaml`.
  3. `Copy-Item mapping\ancien-nom.json mapping\nouveau-nom.json` puis `Remove-Item mapping\ancien-nom.json`.
  4. `python scripts/backfill.py --repo nouveau-nom` (le fichier de mappage empêche les doublons).
  5. `python scripts/deploy_caller_workflows.py --repo nouveau-nom --strategy main`.
  6. Commiter et pousser `vw-jira-sync` vers main.
- Renommer le fichier de mappage *avant* de relancer le backfill — l'ordre est important.
À ne pas faire :
- Exécuter le backfill avec le nouveau nom avant d'avoir renommé le fichier de mappage — cela créera des problèmes Jira en double.
- Supprimer l'ancien fichier de mappage sans l'avoir copié au préalable.
- Supposer que la redirection automatique de GitHub couvre les étiquettes Jira — les étiquettes intègrent le nom du dépôt et doivent être maintenues cohérentes via le backfill.
Terminé quand :
- `config.yaml` utilise le nouveau nom, le fichier de mappage est renommé, le backfill est complété sans erreur, le workflow appelant est redéployé, vw-jira-sync est poussé.
- Guide complet : `vaultwares-docs/docs-content/operations/jira-sync.mdx`
