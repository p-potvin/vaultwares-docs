# Sync GitHub → Jira

Synchronisation unidirectionnelle de GitHub vers Jira Cloud. Réside dans `p-potvin/vw-jira-sync`.

**Modèle de déclenchement préféré :** Webhooks GitHub → `https://hooks.vaultwares.ca/github` → `vw-deployd`
→ exécution de `vw-jira-sync/scripts/live_sync.py` sur `vaultwares-1` (à l'intérieur du tailnet).

## Architecture

```
GitHub event (push / pull_request / review / comment)
        │  (signed webhook)
        ▼
hooks.vaultwares.ca/github  (nginx)
        │  proxy
        ▼
vw-deployd (webhook receiver)
        │  writes event JSON to disk
        │  sets: GITHUB_EVENT_NAME + GITHUB_EVENT_PATH
        ▼
vw-jira-sync/scripts/live_sync.py
        ├─ pull_request                 → upsert Task, transition status
        ├─ pull_request_review          → add review summary comment
        ├─ pull_request_review_comment  → add inline comment
        ├─ issue_comment (on PR)        → add comment
        └─ push (default branch)
              ├─ commit belongs to a PR → comment on PR Task
              └─ direct commit          → create individual Task
```

Chaque dépôt GitHub est lié à exactement un projet Jira. Les tâches sont créées directement dans le
projet — pas d'Épopées (Epics), pas de hiérarchie parentale.

## Types d'incidents Jira

| Événement GitHub | Type d'incident Jira | Motif d'étiquette (Label pattern) |
|---|---|---|
| Pull request | Tâche (Task) | `gh-pr-{owner}-{repo}-{number}` |
| Commit direct (push, pas dans une PR) | Tâche (Task) | `gh-commit-{owner}-{repo}-{sha12}` |

Les étiquettes rendent chaque opération idempotente : les scripts de synchronisation recherchent par étiquette JQL avant de créer.

## Cartographie du statut des PR

| État GitHub | Statut Jira |
|---|---|
| `open` | En cours (In Progress) |
| `draft` | À faire (To Do) |
| `merged` | Terminé (Done) |
| `closed` | Terminé (Done) |

## Secrets

Pour le modèle VPS déclenché par webhook, les secrets résident sur le VPS (et non dans chaque dépôt) :

- `VW_GITHUB_WEBHOOK_SECRET` (utilisé par `vw-deployd` pour vérifier les signatures des webhooks)
- Jeton Jira pour `vw-jira-sync` (`JIRA_TOKEN` ou `JIRA_TOKEN_FILE`)
- Jeton API GitHub pour `vw-jira-sync` (`GITHUB_TOKEN` préféré, ou `GH_TOKEN`)

Ne jamais committer ces secrets. Stockez les valeurs dans VaultWarden (`https://warden.vaultwares.ca`, uniquement sur le tailnet)
et provisionnez-les au VPS via les fichiers d'environnement `/etc/*`.

## Configuration GitHub (par dépôt)

Dépôt → Settings → Webhooks → Ajouter un webhook :

- URL de la charge utile (Payload URL) : `https://hooks.vaultwares.ca/github`
- Type de contenu (Content type) : `application/json`
- Secret : même valeur que `VW_GITHUB_WEBHOOK_SECRET` sur le VPS
- Événements (Events) : sélectionner
  - `push`
  - `pull_request`
  - `pull_request_review`
  - `pull_request_review_comment`
  - `issue_comment`

## Ajout d'un nouveau dépôt

1. Créez un projet Jira (géré par l'équipe, Kanban) avec une clé de 2 à 10 caractères en majuscules.
2. Ajoutez la cartographie dans `config.yaml` :

```yaml
repo_project_keys:
  my-new-repo: MYNR   # ← new entry

repos:
  - my-new-repo       # ← new entry
```

3. Si le dépôt appartient à une autre organisation GitHub, ajoutez-le également à `repo_owners` :

```yaml
repo_owners:
  my-new-repo: OtherOrg
```

4. Ajoutez le webhook GitHub (modèle préféré) :

- Dépôt → Settings → Webhooks → Ajouter un webhook (voir « Configuration GitHub » ci-dessous).

5. Exécutez le remplissage initial (backfill) :

```bash
python scripts/backfill.py --repo my-new-repo
```

6. Optionnel (légacy) : déployez le workflow d'appel vers le nouveau dépôt :

```bash
python scripts/deploy_caller_workflows.py --repo my-new-repo --strategy main
```

## Ajout d'une nouvelle organisation GitHub

1. Ajoutez les dépôts de l'organisation à `repo_owners` dans `config.yaml` :

```yaml
repo_owners:
  repo-name: NewOrg
```

2. Ajoutez chaque dépôt à `repo_project_keys` et `repos` normalement.
3. Assurez-vous que l'outil de ligne de commande `gh` est authentifié pour la nouvelle organisation
   (`gh auth status` — doit montrer l'organisation comme accessible).

## Procédure de renommage de dépôt

Lorsqu'un dépôt GitHub est renommé, la clé de projet Jira et tous les incidents existants restent inchangés.
Seules les étiquettes (qui contiennent le nom du dépôt) et le fichier de cartographie doivent être mis à jour.

**Étape par étape :**

1. Renommez le dépôt sur GitHub (Settings → Rename).

2. Mettez à jour `config.yaml` dans `vw-jira-sync` :

```yaml
repo_project_keys:
  new-repo-name: SAME    # ← update key (was old-repo-name)
  # old-repo-name: SAME  # ← remove this line

repos:
  - new-repo-name         # ← update (was old-repo-name)
```

3. Renommez le fichier de cartographie pour que les incidents existants ne soient pas recréés :

```powershell
Copy-Item "mapping\old-repo-name.json" "mapping\new-repo-name.json"
Remove-Item "mapping\old-repo-name.json"
```

4. Réexécutez le remplissage (backfill) avec le nouveau nom pour étiqueter tout nouvel incident avec l'étiquette actuelle
   (les incidents existants conservent leurs anciennes étiquettes et ne seront pas dupliqués — le fichier de cartographie
   empêche cela) :

```bash
python scripts/backfill.py --repo new-repo-name
```

5. Commit et push vers `vw-jira-sync` main.

6. Déployez le workflow d'appel vers le dépôt renommé (GitHub renomme le dépôt, mais
   le chemin du fichier de workflow reste le même — si il était déjà présent, un nouveau déploiement
   assure que le fichier est à jour) :

```bash
python scripts/deploy_caller_workflows.py --repo new-repo-name --strategy main
```

## Référence du script de remplissage (Backfill)

```bash
# All repos
python scripts/backfill.py

# Single repo
python scripts/backfill.py --repo vault-flows

# Dry-run (print what would be created, no writes)
python scripts/backfill.py --repo vault-flows --dry-run
```

Le remplissage est idempotent. La progression est mise à jour dans `mapping/{repo}.json`
(format : `{"prs": {"42": "PROJ-7"}, "commits": {"abc123def456": "PROJ-8"}}`).

## Déploiement des workflows d'appel (Legacy : GitHub Actions)

```bash
# Single repo, push directly to main
python scripts/deploy_caller_workflows.py --repo my-repo --strategy main

# All repos, open PRs (default)
python scripts/deploy_caller_workflows.py --strategy pr

# All repos, push to main
python scripts/deploy_caller_workflows.py --strategy main
```

## Création de projets Jira en masse

```bash
# Dry run — prints what would be created
python scripts/create_projects.py --dry-run

# Create only for a specific repo
python scripts/create_projects.py --repo my-new-repo

# Create all missing projects
python scripts/create_projects.py
```

## Dépannage

### `startup_failure` dans GitHub Actions (Legacy)

**Cause :** Le workflow d'appel legacy possède un bloc `permissions:` qui entre en conflit avec
le déclenchement de workflow réutilisable sur les comptes personnels.

**Correction :** Supprimez le bloc `permissions:` de `.github/workflows/jira-sync.yml`
dans le dépôt concerné, ou redéployez avec le modèle actuel (qui ne contient pas de
bloc `permissions:`).

### Synchronisation non exécutée par webhook (Modèle préféré)

- Confirmez que `https://hooks.vaultwares.ca/health` retourne 200.
- Confirmez que `https://hooks.vaultwares.ca/github` reçoit des livraisons (vérifiez `/var/log/vw-deployd.log`).
- Assurez-vous que `/etc/vw-deployd/config.yml` contient `jira_sync.enabled: true`.
- Assurez-vous que `vw-jira-sync` existe sur `vaultwares-1` (chemin par défaut : `/opt/vw-jira-sync`).
- Assurez-vous que `vw-jira-sync` dispose des secrets suivants :
  - `JIRA_TOKEN` ou `JIRA_TOKEN_FILE`
  - `GITHUB_TOKEN` (préféré) ou `GH_TOKEN`

### Jira 401 / "Vous n'avez pas la permission de créer des incidents"

**Cause :** Le jeton Jira est corrompu ou expiré (`JIRA_TOKEN` ou `JIRA_TOKEN_FILE`).

**Correction :** Reprovisionnez le jeton sur le VPS (préféré), ou redistribuez les secrets si
vous utilisez encore le modèle GitHub Actions legacy.

### Incidents Jira dupliqués après un renommage

**Cause :** Le fichier de cartographie n'a pas été renommé avant de réexécuter le remplissage, de sorte que le script
n'a pas reconnu les incidents déjà synchronisés.

**Correction :**
1. Supprimez les doublons dans Jira (REST DELETE ou UI).
2. Restaurez le fichier de cartographie depuis git, renommez-le correctement, puis réexécutez le remplissage.

### Erreur de remplissage sur un PR spécifique

**Cause :** Incident GitHub supprimé/restreint ou échec réseau lors de la récupération des commentaires.

**Comportement :** Journalisé comme `WARN` et ignoré — cela n'arrête pas l'exécution. Réexécutez
le remplissage pour le dépôt ; le mécanisme de point de contrôle (checkpointing) signifie que les incidents déjà créés ne seront pas
dupliqués.
