## Deployment and alert note

Every repo in this table must follow the same control-plane model:

- GitHub is only an event source (webhook)
- deploy/test/probe jobs happen inside the tailnet on the configured project
  target host
- **no GitHub Actions deploys**
- alert severity is determined by the Health Ledger alert profile, not by the
  host that receives the webhook
- local Ollama/Gemma remediation is allowed only for explicitly permitted
  non-production Low flows
- `vaultwares-api` owns stats/ledger telemetry DB access, batching ingest,
  auth, normalized monitor APIs, and future business rules. Static dashboards
  such as `stats.vaultwares.ca` must not read source ledger files or Postgres
  directly.
- `GET /monitor/services` reads the fleet rollup assembled from Greencloud,
  OVH, and Clopeux evidence. Every inventory entry has a configured probe;
  `unmonitored` now means configuration failure rather than an accepted
  inventory state. The response includes contributing locations, confirmation
  count, and host-heartbeat diagnostics when available.
- VaultWares apps pass request correlation IDs as the `correlationId` query
  parameter, not as the primary header contract. VaultWares-generated IDs use
  `vw_{3-or-4-char Jira code}_c{7 hex chars}`, for example
  `vw_API_c123abc4`; non-VaultWares products may use their own prefix, for
  example `pk_PKT_c123456h`. The same correlation ID should persist from
  client request creation until the request completes or fails at the API
  boundary. Streams, downloads, and other continuous connections may create a
  child ID for the long-lived transfer.
- `vaultwares-api` request logging defaults to important events only: blocked
  requests, crashes, HTTP errors, and slow requests. Routine successful calls
  are intentionally suppressed to keep Kiwi volume usable. API syslog records
  must include source/client context in addition to the VPS proxy peer address.
- Vault-flows ComfyUI jobs submitted through `vaultwares-api` are serialized:
  one ComfyUI workflow job may execute at a time, regardless of source IP.
  Asynchronous queue clients can provide a completion callback URL; callbacks
  are limited to loopback, private, or tailnet hosts unless explicitly allowed.

Source of truth: `operations/deployment-flow`.

persistent config and data under `/srv/vw-media/`. Greencloud `dnsmasq` maps the
service hostnames below to `100.73.93.84`; Greencloud nginx then proxies to OVH
over the tailnet.

Mullvad VPN routing for this stack is opt-in only. The staged gateway lives in
`/opt/vw-media-stack/docker-compose.mullvad.yml` as `mullvad-gateway`. **DO NOT** run `docker compose up -d` directly to rebuild the stack if VPN routing is required.

Always use the dedicated wrapper script:
```bash
/opt/vw-media-stack/bin/mullvad-compose all-up
```

## Residential egress proxy on `brume2`

Tube-site source CDNs (`fullvideos.xxx`, `1porn.tv`, `pornxp.*`) IP-block both
OVH and Greencloud datacenter ranges via Cloudflare and bind their signed MP4
URLs to the requester IP. `shared-tube` `/api/stream/[slug]` fetches through a
residential proxy on Brume2 so both legs egress the same residential IP.

| Surface | Listener / unit | Role |
