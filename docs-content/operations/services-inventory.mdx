## Deployment and alert note

Every repo in this table must follow the same control-plane model:

- GitHub is only an event source (webhook)
- deploy/test/probe jobs happen inside the tailnet on the configured project
  target host
- **no GitHub Actions deploys**
- alert severity is determined by the Health Ledger alert profile, not by the
  host that receives the webhook
- local Ollama/Gemma remediation is allowed only for explicitly permitted
  non-production Low flows
- `vaultwares-api` owns stats/ledger telemetry DB access, batching ingest,
  auth, normalized monitor APIs, and future business rules. Static dashboards
  such as `stats.vaultwares.ca` must not read source ledger files or Postgres
  directly.
- `GET /monitor/services` reads the fleet rollup assembled from Greencloud,
  OVH, and Clopeux evidence. Every inventory entry has a configured probe;
  `unmonitored` now means configuration failure rather than an accepted
  inventory state. The response includes contributing locations, confirmation
  count, and host-heartbeat diagnostics when available.
- VaultWares apps pass request correlation IDs as the `correlationId` query
  parameter, not as the primary header contract. VaultWares-generated IDs use
  `vw_{3-or-4-char Jira code}_c{7 hex chars}`, for example
  `vw_API_c123abc4`; non-VaultWares products may use their own prefix, for
  example `pk_PKT_c123456h`. The same correlation ID should persist from
  client request creation until the request completes or fails at the API
  boundary. Streams, downloads, and other continuous connections may create a
  child ID for the long-lived transfer.
- `vaultwares-api` request logging defaults to important events only: blocked
  requests, crashes, HTTP errors, and slow requests. Routine successful calls
  are intentionally suppressed to keep Kiwi volume usable. API syslog records
  must include source/client context in addition to the VPS proxy peer address.
- Vault-flows ComfyUI jobs submitted through `vaultwares-api` are serialized:
  one ComfyUI workflow job may execute at a time, regardless of source IP.
  Asynchronous queue clients can provide a completion callback URL; callbacks
  are limited to loopback, private, or tailnet hosts unless explicitly allowed.

Source of truth: `operations/deployment-flow`.

## Tailnet media surfaces on `vps-ovhcloud`

These services are intended for tailnet-only operator use. Docker-backed services
live on `vps-ovhcloud` in `/opt/vw-media-stack/docker-compose.yml` with
persistent config and data under `/srv/vw-media/`. Greencloud `dnsmasq` maps the
service hostnames below to `100.73.93.84`; Greencloud nginx then proxies to OVH
over the tailnet. Client DNS must include the matching Tailscale split-DNS/NRPT
names, otherwise public wildcard DNS resolves these hosts to Greencloud's public
address and the tailnet-only proxy path is bypassed.
Greencloud serves HTTPS for these hostnames with the
`/etc/letsencrypt/live/media.vaultwares.ca/` SAN certificate.

Greencloud also has explicit `listen 100.73.93.84:443 ssl;` entries for
tailnet service names such as `docs.vaultwares.ca`, `hooks.vaultwares.ca`,
`warden.vaultwares.ca`, `monitor.vaultwares.ca`, `stats.vaultwares.ca`, and
`mcp.vaultwares.ca`. Keep those listeners when adding private hostnames; without
them, nginx can select the media certificate on tailnet-originated requests.

Mullvad VPN routing for this stack is opt-in only. The staged gateway lives in
`/opt/vw-media-stack/docker-compose.mullvad.yml` as `mullvad-gateway`. **DO NOT** run `docker compose up -d` directly to rebuild the stack if VPN routing is required, as this will strand services on the default network and break communication with the Arr stack.

Instead, always use the dedicated wrapper script:
```bash
/opt/vw-media-stack/bin/mullvad-compose all-up
```
This script seamlessly boots the `mullvad-gateway` and automatically applies the overrides (e.g. `docker-compose.mullvad.qbittorrent.yml`) to move `qbittorrent` and `jackett` into `network_mode: service:mullvad-gateway`.

The gateway configures itself via `.env.mullvad` using Gluetun's native Mullvad API (e.g., setting `MULLVAD_SERVER_CITIES=Montreal` and `MULLVAD_SERVER_HOSTNAMES=ca-mtr-wg-002`). Profile rotation is also supported via `/opt/vw-media-stack/bin/mullvad-rotate-profile` using dedicated VPS profiles under `/opt/vw-media-stack/mullvad-profiles/`. The gateway keeps `100.64.0.0/10` outside the tunnel so tailnet access, including `100.67.25.118`, stays reachable. Do not add host-wide WireGuard routes for this stack.

## Residential egress proxy on `brume2`

Tube-site source CDNs (`fullvideos.xxx`, `1porn.tv`, `pornxp.*`) IP-block both
OVH and Greencloud datacenter ranges via Cloudflare and bind their signed MP4
URLs to the requester's IP. `shared-tube`'s `/api/stream/[slug]` route fetches
both the source page and the byte-range MP4 through a residential proxy on the
home-network Brume2 router so both legs egress the same residential IP.

| Surface | Listener / unit | Role |
|---|---|---|
| `tinyproxy` | `http://100.114.136.28:8888` (tailnet-only, allowlisted to `100.64.0.0/10`) | HTTP/CONNECT forward proxy on `brume2` (OpenWrt, GL-MT2500). Egresses the home WAN (`74.57.201.206`, Bell/Videotron). Read `operations/residential-egress-proxy` for the full architecture, env vars, and provisioning steps. |

Both `vps-ovhcloud` and `greencloud-vps` carry `STREAM_OUTBOUND_PROXY=http://100.114.136.28:8888` in the relevant `shared-tube` env files for `fxv`, `oneporn`, and `sexyprn` so the active tube apps share the same exit. The proxy is a single point of failure for video playback; see the troubleshooting section of the egress-proxy page if streams 502.
