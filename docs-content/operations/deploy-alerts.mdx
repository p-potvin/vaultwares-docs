# Deploy alerts (Private)

Last updated: Sun, 28 Jun 2026 10:24 America/Toronto.

> Status: legacy alert writers were disabled on `greencloud-vps` on
> 2026-06-03. This page remains as a reference for the old scripts and for
> rollback/forensics. The replacement direction is Health Ledger plus Alarm
> Joker and Token Joker.

Historically, the webhookd dispatcher on `greencloud-vps` ran two notification surfaces:

1. **On-error notify** — fires the moment a target deploy script exits non-zero.
2. **Bad-signature watcher** — a 5-minute systemd timer that opens a single
   consolidated issue when a cluster of `deny: bad_signature` lines appears
   in the webhookd log (the runner is silently rejecting GitHub deliveries).

Both surfaces filed a GitHub issue and mentioned `@copilot`. They were disabled
because the path was GitHub-only, intermittently broken by token scope/auth
state, and too noisy for the new Health Ledger alert model.

Related:
- `operations/deployment-flow` (canonical flow)
- `operations/webhook-secret-rotation` (rotation runbook)
- `operations/services-inventory` (single table of URLs)

| Path | Owner / mode | Role |
|---|---|---|
| `/usr/local/bin/vw-deploy-notify.sh`     | `root:vwdeploy 0750` | Legacy script. Previously invoked by `vw-webhookd`; now kept only for reference. |
| `/usr/local/bin/vw-deny-watch.sh`        | `root:root 0750`     | Legacy script. Timer disabled on 2026-06-03. |
| `/var/log/vw-deploy-notify.log`          | `root:vwdeploy 0660` | Append-only log for both scripts. Records `notify fired:` + `issue opened:` (or the failure reason). Never contains the token. |

| `/etc/vw-webhookd/env`                   | `root:vwdeploy 0640` | Holds `VW_GITHUB_WEBHOOK_SECRET` and may still hold legacy `GH_TOKEN` for reference scripts. Prom-King deploy git auth does not use it. **No CRLF** if edited. |
| `/etc/vw-github-app/`                    | `root:vwdeploy 0640/0750` | Active Prom-King deploy git credential on greencloud: App config and private key consumed by `/usr/local/bin/vw-github-app-token`. |

## Replacement direction

Health Ledger owns the active direction:

- project routing is a lookup table, not a host-based production decision,
- alert profile determines Low/Medium/High/Critical behavior,
- local Ollama/Gemma remediation is allowed only for explicitly permitted
  non-production Low flows,
- production incidents use Alarm Joker and must not depend on local Ollama,
- Token Joker tracks credential age and health, including the Prom-King GitHub
  App private key used for deploy git access.

## How the on-error path fires

```text
git push (main) → vw-webhookd dispatches deploy script
  → deploy script exits != 0
  → webhookd writes "exit=N" then "notify: …" to /var/log/vw-webhookd.log
  → webhookd execs bash -lc /usr/local/bin/vw-deploy-notify.sh
       env = os.environ + {VW_NOTIFY_TITLE, VW_NOTIFY_BODY}
  → notify script sources /etc/vw-webhookd/env to pick up GH_TOKEN
  → `gh issue create --repo <alerts repo> --label deploy-failure …`
  → URL written to /var/log/vw-deploy-notify.log
```

This path is legacy-only while `notifications.on_error_command` is `/bin/true`.
Do not make Prom-King deploys depend on this PAT.

## How the bad-signature watcher fires

```text
systemd timer (5 min) → /usr/local/bin/vw-deny-watch.sh
  → reads `/var/lib/vw-webhookd/last-deny-mark` (last alerted timestamp)
  → counts `deny: bad_signature` lines in /var/log/vw-webhookd.log newer than the mark
  → if count ≥ THRESHOLD (default 3): file ONE issue, advance the mark
  → otherwise: no-op
```

## GitHub credentials

Prom-King deploy repository access is GitHub App based:

- App ID `4165781`.
- greencloud and OVH store App config at `/etc/vw-github-app/app.env`.
- greencloud and OVH store the private key at
  `/etc/vw-github-app/private-key.pem`.
- `/usr/local/bin/vw-github-app-token Prom-King` mints and caches an
  installation token.
- `/var/www/deploy-scripts/git-askpass-vw-gh-token.sh` feeds that installation
  token to HTTPS git when `VW_GITHUB_OWNER=Prom-King`.

Rotate the App private key when Prom-King deploy repository access changes.
The old deploy-alert PAT path is only for disabled legacy issue writers.

## Common mistakes (these have bitten us already)

| Symptom | Cause | Fix |
|---|---|---|
| `gh: invalid header field value for "Authorization"` from a legacy alert script | Env file was saved with CRLF; `\r` got embedded in `GH_TOKEN`. | `sed -i 's/\r//g' /etc/vw-webhookd/env`. This does not affect Prom-King deploy git auth. |
| `git ls-remote` returns 404 for `Prom-King/*` with App askpass | App is not installed on the org/repo, `VW_GITHUB_OWNER` is missing, or the helper cannot read `/etc/vw-github-app/private-key.pem`. | Confirm App installation, export `VW_GITHUB_OWNER=Prom-King`, check helper/key permissions, then run `git ls-remote https://github.com/Prom-King/<repo>.git HEAD`. |
| `Permission denied` from inside the script when webhookd invokes it | Script perms are `root:root 0750`; `vwdeploy` can't execute. | `chown root:vwdeploy /usr/local/bin/vw-deploy-notify.sh && chmod 0750 …` |
| Notify never fires for failed deploys | `vw_jira_sync` exited non-zero and the old webhookd shortcircuited before reaching the deploy dispatch. | Already patched: `_notify_on_error` is invoked from the deploy path; `vw_jira_sync` failures are now logged but non-blocking. See `operations/deployment-flow#vw_jira_sync-non-blocking`. |

## Verifying the alert chain end-to-end

1. Force a fake failure: push an empty commit that the deploy script will
   process; flip a known-good file to a bad value momentarily; revert. Or
   on the box, run `sudo -u vwdeploy bash -lc 'VW_AFTER=<sha> /var/www/deploy-scripts/deploy-tube-sites.sh; echo exit=$?'`.
2. Check `/var/log/vw-webhookd.log` for `notify: /usr/local/bin/vw-deploy-notify.sh`.
3. Check `/var/log/vw-deploy-notify.log` for `issue opened: https://github.com/…`.
4. Open the URL — verify `@copilot` is mentioned and the labels `deploy-failure`, `automated-alert` are set.
