# Deploy alerts (Private)

The webhookd dispatcher on `greencloud-vps` runs two notification surfaces:

1. **On-error notify** — fires the moment a target deploy script exits non-zero.
2. **Bad-signature watcher** — a 5-minute systemd timer that opens a single
   consolidated issue when a cluster of `deny: bad_signature` lines appears
   in the webhookd log (the runner is silently rejecting GitHub deliveries).

Both surfaces file a GitHub issue and mention `@copilot` so the Copilot
assignee can pick up the fix without a human round-trip.

- `operations/deployment-flow` (canonical flow)

## Files on `greencloud-vps`

| Path | Owner / mode | Role |
|---|---|---|
| `/usr/local/bin/vw-deploy-notify.sh`     | `root:vwdeploy 0750` | Invoked by `vw-webhookd` via `notifications.on_error_command`. Parses `VW_NOTIFY_TITLE` (`Command failed: <owner>/<repo>`), opens a GitHub issue tagging `@copilot`. |
| `/usr/local/bin/vw-deny-watch.sh`        | `root:root 0750`     | Run by systemd every 5 min. Diffs `/var/log/vw-webhookd.log` against `/var/lib/vw-webhookd/last-deny-mark` and files **one** consolidated issue when the new `bad_signature` count crosses `THRESHOLD` (default `3`). |
| `/var/log/vw-deploy-notify.log`          | `root:vwdeploy 0660` | Append-only log for both scripts. Records `notify fired:` + `issue opened:` (or the failure reason). Never contains the token. |
| `/var/lib/vw-webhookd/last-deny-mark`    | `root:root 0600`     | Single line containing the timestamp of the most recently alerted `bad_signature` row. |
| `/etc/vw-webhookd/env`                   | `root:vwdeploy 0640` | Holds `VW_GITHUB_WEBHOOK_SECRET` **and** `GH_TOKEN`. The notify script sources this file when run. **No CRLF** — see "Common mistakes" below. |
| `/etc/systemd/system/vw-deny-watch.timer` + `.service` | `root 0644` | The 5-minute timer + the oneshot service that runs `vw-deny-watch.sh`. |
| `/etc/vw-webhookd/config.yml`            | `root 0644`          | `notifications.on_error_command: "/usr/local/bin/vw-deploy-notify.sh"` wires the on-error path. |

## How the on-error path fires

```text
git push (main) → vw-webhookd dispatches deploy script
  → deploy script exits != 0
  → webhookd writes "exit=N" then "notify: …" to /var/log/vw-webhookd.log
  → webhookd execs bash -lc /usr/local/bin/vw-deploy-notify.sh
       env = os.environ + {VW_NOTIFY_TITLE, VW_NOTIFY_BODY}
  → notify script sources /etc/vw-webhookd/env to pick up GH_TOKEN
  → `gh issue create --repo <alerts repo> --label deploy-failure …`
  → URL written to /var/log/vw-deploy-notify.log
```

## How the bad-signature watcher fires

```text
systemd timer (5 min) → /usr/local/bin/vw-deny-watch.sh
  → reads `/var/lib/vw-webhookd/last-deny-mark` (last alerted timestamp)
  → counts `deny: bad_signature` lines in /var/log/vw-webhookd.log newer than the mark
  → if count ≥ THRESHOLD (default 3): file ONE issue, advance the mark
  → otherwise: no-op
```

## GH_TOKEN scope + alerts repo choice

The token in `/etc/vw-webhookd/env` is a **fine-grained PAT** with `issues: write`
on **selected repos**. As of 2026-05-30 the active token is scoped to
`p-potvin/*` only; **`Prom-King/*` repos return 404 on `gh issue create`**.

Until the token is regenerated with Prom-King access, both scripts default
to the canonical alerts repo:

- **`p-potvin/vaultwares-docs`** — same place the runbooks live, so Copilot
  picks up the fix near the doc trail.

The issue title carries the failing repo + commit, so triage stays simple
even when alerts pile into one tracking repo.

## Token rotation (every 30 days)

The token expires fast. A daily systemd timer warns 14 days ahead and
opens a rotation issue 7 days ahead so we don't go dark.

| Path | Role |
|---|---|
| `/usr/local/bin/vw-token-expiry-watch.sh` | Calls `GET /user` with the token, reads `github-authentication-token-expiration` response header, files an issue if days-left ≤ `WARN_DAYS` (default 7). |
| `/etc/systemd/system/vw-token-expiry-watch.timer` | `OnCalendar=daily` |
| `/etc/systemd/system/vw-token-expiry-watch.service` | oneshot |

Rotation procedure (10 min):

1. Generate a new fine-grained PAT on GitHub. Account: `p-potvin`. Expiration: 60 days. Resource access: **add every `Prom-King/*` repo we deploy + `p-potvin/*` repos** with the webhook. Permissions: **Repository → Issues → Read and write**.
2. Save the PAT alongside the existing one at
   `C:\Users\Administrator\Desktop\Github Repos\.access\gh-finegrained-token-deployments-alerts.txt`
   (overwrite — that path is the canonical operator copy).
3. On `greencloud-vps`:
   ```bash
   sed -i 's/\r//g' /etc/vw-webhookd/env   # strip CRLF, just in case
   $EDITOR /etc/vw-webhookd/env              # replace GH_TOKEN=... line
   chown root:vwdeploy /etc/vw-webhookd/env
   chmod 0640 /etc/vw-webhookd/env
   ```
   No service restart needed; both alert scripts re-source the env file
   on every invocation.
4. Smoke test:
   ```bash
   sudo -u vwdeploy bash -c '
     . /etc/vw-webhookd/env
     gh api user --jq .login
     gh api repos/Prom-King/tube-sites --jq .full_name
   '
   ```
   First line should print `p-potvin` (or whichever account owns the PAT).
   Second line should print `Prom-King/tube-sites`, **not** `Not Found`.

## Common mistakes (these have bitten us already)

| Symptom | Cause | Fix |
|---|---|---|
| `gh: invalid header field value for "Authorization"` | Env file was saved with CRLF; `\r` got embedded in `GH_TOKEN`. | `sed -i 's/\r//g' /etc/vw-webhookd/env` |
| `gh: Not Found (HTTP 404)` on `Prom-King/*` even though the token works on `p-potvin/*` | Fine-grained PAT was scoped to one org only. | Generate a new PAT with both orgs in the resource list. Or keep alerts pointing at `p-potvin/vaultwares-docs`. |
| `Permission denied` from inside the script when webhookd invokes it | Script perms are `root:root 0750`; `vwdeploy` can't execute. | `chown root:vwdeploy /usr/local/bin/vw-deploy-notify.sh && chmod 0750 …` |
| Notify never fires for failed deploys | `vw_jira_sync` exited non-zero and the old webhookd shortcircuited before reaching the deploy dispatch. | Already patched: `_notify_on_error` is invoked from the deploy path; `vw_jira_sync` failures are now logged but non-blocking. See `operations/deployment-flow#vw_jira_sync-non-blocking`. |

## Verifying the alert chain end-to-end

1. Force a fake failure: push an empty commit that the deploy script will
   process; flip a known-good file to a bad value momentarily; revert. Or
   on the box, run `sudo -u vwdeploy bash -lc 'VW_AFTER=<sha> /var/www/deploy-scripts/deploy-tube-sites.sh; echo exit=$?'`.
2. Check `/var/log/vw-webhookd.log` for `notify: /usr/local/bin/vw-deploy-notify.sh`.
3. Check `/var/log/vw-deploy-notify.log` for `issue opened: https://github.com/…`.
4. Open the URL — verify `@copilot` is mentioned and the labels `deploy-failure`, `automated-alert` are set.
