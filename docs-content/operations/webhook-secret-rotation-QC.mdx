# Rotation du secret webhook (`VW_GITHUB_WEBHOOK_SECRET`)

Voici la **checklist** officielle pour tourner le secret qui sert à vérifier les signatures des webhooks GitHub.

Ça doit être fait **de bout en bout** :

- `vw-webhookd` (le récepteur) doit utiliser le nouveau secret
- chaque webhook GitHub qui pointe vers `https://hooks.vaultwares.ca/github` doit utiliser le même nouveau secret

Si tu tournes juste un côté, tu vas voir des `401` ou des `bad_signature` dans les logs et les déploiements vont arrêter.

## Avant de commencer (1 minute)

- Confirme que personne ne déploie en ce moment.
- Confirme que tu as accès au tailnet vers `greencloud-vps` (`100.73.93.84`).
- Confirme que `https://hooks.vaultwares.ca/health` retourne `200`.

## Qu’est-ce qu’on tourne (définition simple)

Pense à ce secret-là comme un **tampon partagé** :

- GitHub “tamponne” chaque message webhook avec ce secret.
- `vw-webhookd` vérifie le tampon.
- Si ça match pas → le message est refusé.

## Checklist (dans cet ordre)

### 1) Générer un nouveau secret

- Génère une nouvelle valeur hex de 64 caractères (32 bytes).
- Stocke-la dans **VaultWarden** sous le nom `VW_GITHUB_WEBHOOK_SECRET`.
- Ne colle pas la valeur dans des docs, dans le chat, dans un commit, ni dans des logs.

### 2) Installer le nouveau secret sur `greencloud-vps`

- Mets à jour le fichier d’environnement de `vw-webhookd` :
  - Fichier : `/etc/vw-webhookd/env`
  - Clé : `VW_GITHUB_WEBHOOK_SECRET=...`
- Assure-toi qu’il n’y a **pas d’espaces** avant/après.
- Permissions serrées (root, `0600`).

### 3) Redémarrer `vw-webhookd`

- Redémarre le service systemd : `vw-webhookd.service`
- Vérifie qu’il roule : `systemctl is-active vw-webhookd` → `active`

### 4) Mettre à jour les webhooks GitHub de tous les repos

Pour chaque repo qui déploie via `vw-webhookd` :

- Trouve le webhook dont l’URL est `https://hooks.vaultwares.ca/github`
- Mets à jour son secret pour matcher le nouveau `VW_GITHUB_WEBHOOK_SECRET`
- Garde les événements cohérents (minimum : `push`, + événements PR si utilisés ailleurs)

### 5) Vérifier (obligatoire)

Fais **les deux** :

1) Ping chaque webhook (la livraison GitHub “ping” doit être HTTP `200`).
2) Déclenche un vrai déploiement :
   - push un commit vide sur un repo à faible risque, ou
   - roule le script de déploiement manuellement sur `greencloud-vps`.

Ensuite, confirme :

- `/var/log/vw-webhookd.log` montre un `push: repo=...` et la commande de déploiement s’est exécutée.
- il n’y a **pas** de nouvelles lignes `deny: bad_signature` après la rotation.

## Pannes fréquentes (diagnostic rapide)

| Symptom | Cause probable | Fix |
|---|---|---|
| Livraison GitHub = `401` | secret pas pareil | mettre à jour le secret du webhook du repo |
| `vw-webhookd.log` dit `deny: bad_signature` | secret pas pareil ou espaces | vérifier `/etc/vw-webhookd/env` (pas d’espaces/newlines) |
| Deploy fail `dubious ownership` | `safe.directory` manquant | ajouter `safe.directory` pour le user de déploiement |
| Deploy peut pas écrire dans `/var/www/...` | mauvais owner/permissions | donner le web root au user de déploiement |

## Notes (hygiène ops)

- `vw-webhookd` roule sous l’usager `vwdeploy` sur `greencloud-vps`.
- Les scripts de déploiement doivent être exécutables par `vwdeploy`, et les dossiers cibles (repo checkout + web root) doivent être écrivable par `vwdeploy`.
