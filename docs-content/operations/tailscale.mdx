# Tailscale (Private)

## Policy goals

- Tailnet is the **default control plane**.
- VPS SSH is **tailnet-only**.
- Services are explicitly allowed by policy (deny-by-default).

## Key rules (current)

- Owner can reach `tag:server` on `22`, `80`, `443`.
- If we use split DNS with `greencloud-vps` as the nameserver, also allow `tag:server` on **DNS** (`53` TCP+UDP).
- Owner can reach `100.71.101.21` on `3389` (RDP) and `100.67.25.118:9001` (VaultWares API).
- `tag:server` can reach `100.67.25.118:9001` (VPS reverse-proxy to the OVH API host).
- For the Health Ledger probe triangle, `tag:server` must also be able to
  reach other `tag:server` nodes on `80` and `443`. Without that, OVH probes
  can resolve Greencloud private names but cannot connect to the Greencloud
  nginx surfaces.
- **SSH dispatch:** the SSH policy (`ssh` block) for `dst: tag:vps-ovhcloud` must
  include `tag:vps-greencloud` in `src`. The deploy dispatcher scripts on
  greencloud SSH `ubuntu@100.67.25.118` over the tailnet to run OVH-side deploys;
  Tailscale-SSH gates that connection, not OpenSSH `authorized_keys`. Missing
  this tag silently breaks **all** OVH-deployed services. See
  `operations/deployment-flow` for the failure signature.
- **Residential egress proxy:** `tag:vps-ovhcloud` and `tag:vps-greencloud` (plus
  owner personal devices) must be able to reach `100.114.136.28:8888` (the
  tinyproxy on Brume2). This is the path `shared-tube /api/stream/*` uses to
  fetch Cloudflare-protected source CDNs from a residential IP. See
  `operations/residential-egress-proxy` for context.

## Exposing internal web services to the tailnet (Split DNS)

Some subdomains, including `stats.vaultwares.ca`, are **tailnet-only** services. The public DNS record returns the VPS public IP, which serves a 403 to all non-tailnet clients. Tailnet nodes resolve the same hostname to the **Tailscale IP** of `greencloud-vps` via a Tailscale DNS override, bypassing the public nginx vhost entirely and hitting a separate internal-only nginx server block.

### How it works

```
Public client  ->  stats.vaultwares.ca  ->  <public IP>  ->  nginx (403, intentional)
Tailnet node   ->  stats.vaultwares.ca  ->  100.x.x.x    ->  nginx (internal vhost, 200)
```

### Step 1 -- Tailscale DNS override

In the [Tailscale admin console](https://login.tailscale.com/admin/dns) -> **Nameservers**:

- Add **Restricted / Split DNS** entries for each private hostname that must resolve through the tailnet.
- Point each entry to the Tailscale IP of `greencloud-vps` (`100.73.93.84`).
- Keep public names (for example `vaultwares.ca`, `www.vaultwares.ca`, `flows.vaultwares.ca`) out of this exact-host override unless you intentionally want tailnet clients to use the VPS tailnet address for those public services.

`api.vaultwares.ca` is intentionally excluded from the Greencloud split-DNS
set. It resolves to `vps-ovhcloud` (`100.67.25.118`) because OVH owns the
VaultWares API and local Postgres runtime. OVH nginx owns `100.67.25.118:443`;
do not enable Tailscale Serve on HTTPS `443` on OVH, because it intercepts
tailnet clients before nginx and breaks `api.vaultwares.ca`.

`greencloud-vps` runs `dnsmasq` on `tailscale0` and answers those names from `/etc/dnsmasq.d/vaultwares-tailnet.conf`. The config uses exact `host-record` entries instead of a wildcard, so it does not take over the whole `vaultwares.ca` zone.

Sun, 28 Jun 2026 12:25: keep explicit `interface=lo` and `interface=tailscale0` lines in the greencloud `dnsmasq` config. Debian starts dnsmasq with `--local-service`; without explicit interfaces, remote tailnet clients can be treated as non-local because `tailscale0` is a `/32`, causing split-DNS timeouts even though UFW and Tailscale ACLs allow port `53`.

If you choose to route the parent zone `vaultwares.ca` to `greencloud-vps`, `dnsmasq` must also forward unknown `vaultwares.ca` names to an external resolver. Do not use the VPS default resolver (`100.100.100.100`) for that parent-zone forwarding path, because Tailscale DNS can route the query back to the same split-DNS target.

> The ACL must allow `tag:server` on port **53** (TCP+UDP) if `greencloud-vps` is the authoritative nameserver. See Key rules above.

### Step 2 -- nginx internal vhost on the Tailscale interface

Add a second `server` block that listens only on the Tailscale IP:

```nginx
server {
    listen 100.x.x.x:443 ssl;
    server_name stats.vaultwares.ca;

    ssl_certificate     /etc/nginx/certs/stats.vaultwares.ca.crt;
    ssl_certificate_key /etc/nginx/certs/stats.vaultwares.ca.key;

    root /var/www/stats.vaultwares.ca;
    index index.html;

    location / {
        try_files $uri $uri/ /index.html;
    }
}
```

Replace `100.x.x.x` with the actual Tailscale IP of `greencloud-vps`. Reload nginx:

```bash
sudo nginx -t && sudo systemctl reload nginx
```

### TLS certificate for the internal vhost

You have two options:

- **`tailscale cert`** -- generates a Let's Encrypt cert valid for the MagicDNS FQDN. Works if the vhost `server_name` matches the MagicDNS name.
- **Existing Let's Encrypt cert** -- if you already have a wildcard or SAN cert for `*.vaultwares.ca`, reuse it here.

### ACL note

No ACL change is needed for the web traffic itself -- the existing rule `owner -> tag:server : [80, 443]` already covers it. The Split DNS entry is the sole gate.

## Tailscale Serve on Clopeux-Desktop

`Clopeux-Desktop` runs Tailscale Serve to expose `stash-win` (Stash media manager) on port `9999` to the tailnet:

```
https://clopeux-desktop.tail1764b8.ts.net (tailnet only)
|-- / proxy http://127.0.0.1:9999
```

This is the only Tailscale Serve endpoint in the fleet. Do not enable Tailscale Serve on `vps-ovhcloud` port `443` -- it intercepts tailnet clients before nginx and breaks `api.vaultwares.ca`.

## Operational notes

- Avoid tagging personal devices unless you intentionally want them to behave like "service accounts".
- Prefer dedicated tags for infrastructure: `tag:server`, `tag:vps-old`, `tag:vps-greencloud`.
- Brume2 (`brumeux`, `100.114.136.28`) is a stock-flashed proxy-only node; it
  no longer runs the home drop-in gateway / AdGuard. Its SSH is Tailscale-SSH
  only (OpenSSH key/password auth is disabled by the GL.iNet firmware when
  Tailscale-SSH is on). See `operations/residential-egress-proxy`.

### SSH session-recording pitfall

If you add `recorder` to any SSH ACL rule without standing up a `tsrecorder`
node first, **every session matching that rule is killed at the protocol
banner** with:

> *This SSH session was terminated by Tailscale because the server could not
> stream session logs to the session recorder.*

This includes the automated dispatcher sessions used by `vw-webhookd`. Symptom:
deploys silently stop firing across the fleet even though the webhook delivers
and the dispatcher script starts. Either provision a recorder or set
`enforceRecorder: false` on the rule (allows session through when the recorder
is unreachable).
