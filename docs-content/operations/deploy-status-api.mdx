# Deploy status API

Last updated: Sun, 05 Jul 2026 03:20 America/Toronto.

Single source of truth for **"is this project built, what version is live, when did the last rebuild finish?"** — the questions that used to require SSHing into greencloud/OVH and tailing a log file.

Feeds the vault-monitor **Deployments** panel (`Services` page) and the shared-tube admin **Marketing** tab's rebuild spinner + version chip.

## Endpoint

```
GET /api/monitor/deploys
```

Router: `vaultwares-api/app/routers/monitor/__init__.py` (function `get_deploys_summary`).

### Query params

| Param | Type | Purpose |
| --- | --- | --- |
| `project` | string, optional | Filter to one project id (`shared-tube`, `vaultwares-api`, ...). Omitted → every project. |
| `site` | string, optional | For multi-site projects (shared-tube's `fxv`/`oneporn`/`sexyprn`), filter the `targets[]` array. |
| `logs` | bool, default `false` | Include the last ~40 lines of the deploy log per project. Expensive; only ask when the panel needs it. |

### Response shape

```json
{
  "as_of": "2026-07-05T02:47:00Z",
  "projects": {
    "shared-tube": {
      "project": "shared-tube",
      "repo_sha": "b4cf85f",
      "shared_version": "0.2.29",
      "log_path": "/var/log/vw-deploy/shared-tube.log",
      "targets": [
        {
          "id": "shared-tube:fxv",
          "site": "fxv",
          "version": "0.1.8",
          "sha": "b4cf85f",
          "systemd": { "unit": "fxv.service", "active": "active", "since": "2026-07-05T02:33:00Z" }
        },
        { "id": "shared-tube:oneporn", "site": "oneporn", "version": "0.1.8", "sha": "b4cf85f", "systemd": { ... } },
        { "id": "shared-tube:sexyprn", "site": "sexyprn", "version": "0.1.8", "sha": "b4cf85f", "systemd": { ... } }
      ],
      "last_build": {
        "phase": "ok",
        "sha": "b4cf85f",
        "site": null,
        "started_at": "2026-07-05T02:30:00Z",
        "finished_at": "2026-07-05T02:33:00Z",
        "duration_s": 180,
        "exit_code": 0,
        "ok": true
      },
      "log_tail": ["… last ~40 lines when logs=true …"]
    },
    "vaultwares-api": {
      "project": "vaultwares-api",
      "api_version": "0.2.8",
      "repo_sha": "299d45c",
      "targets": [ { "id": "vaultwares-api", "site": null, "version": "0.2.8", "sha": "299d45c", "systemd": { "unit": "vaultwares-api.service", "active": "active", "since": "…" } } ],
      "last_build": { ... }
    }
  }
}
```

### Phase values

| Phase | Meaning |
| --- | --- |
| `ok` | Last build finished cleanly. `finished_at` + `ok=true`. |
| `failed` | Last build exited non-zero. Check `log_tail` / `log_path`. |
| `building` | A build is in flight. Detected by the presence of `/var/lib/vw-deploy/<project>.lock` newer than the last recorded `finished_at`. Populated by the deploy script's phase=building pre-write **and** by the rebuild.py in-flight marker. |
| `unknown` | Status file missing or corrupt. Shouldn't happen once the project is onboarded. |

## Data contract — how the JSON gets there

Every deploy script writes `/var/lib/vw-deploy/status/<project>.json` on completion. `/api/monitor/deploys` just reads and normalises those files — **no SSH at request time**.

### Per-project responsibilities

| Project | Runs on | Written by |
| --- | --- | --- |
| `shared-tube` | greencloud (100.73.93.84), OVH mirror (100.67.25.118) | `Prom-King/shared-tube/scripts/deploy-shared-tube.sh` — phase=building on start, phase=ok/failed on trap EXIT. |
| `vaultwares-api` | OVH (100.67.25.118) | `Github Repos/vaultwares-api/scripts/post-deploy.sh` — invoked after `systemctl restart vaultwares-api.service`. |

Both scripts use the same atomic-write pattern (`.tmp` + `os.replace`) so `/monitor/deploys` never sees a torn file.

### Onboarding a new project

1. Add a deploy step to your project's script that atomic-writes `/var/lib/vw-deploy/status/<project>.json` matching the schema in the response section above (minimum fields: `project`, `targets`, `last_build`).
2. If the project has a lock file, place it at `/var/lib/vw-deploy/<project>.lock` so the in-flight detector picks it up.
3. Pre-write a `phase=building` record at start so the panel doesn't show a stale phase while pnpm builds.
4. Confirm `curl -s http://<api-host>/api/monitor/deploys?project=<yours> | jq .projects` returns the new project.
5. (Optional) Extend `vault-monitor/src/features/deploys/DeploysPanel.tsx` if the project has non-default columns.

## Consumers

- **`vault-monitor`** — `src/features/deploys/DeploysPanel.tsx` calls `getDeploys()` (`src/api.ts`) every 30 s and renders one row per (project, target). This is the fleet view.
- **`shared-tube` admin Marketing tab** — `shared/src/admin/Marketing/Settings.tsx` polls this endpoint at 3 s while a rebuild is in flight, 30 s otherwise. The rebuild button waits on `phase` transitions and then probes the live HTML for the just-saved head snippet.

If a third consumer emerges, extend response fields additively; do **not** clone the endpoint. That's the whole point of this doc.
