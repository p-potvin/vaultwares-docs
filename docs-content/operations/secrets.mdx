# Secrets (Private)

## Bootstrap: Vaultwarden (now)

- Purpose: store operational secrets immediately (tokens, DB credentials, renewal notes).
- Location: running on `vaultwares-1` and reachable over Tailnet only.
- Endpoint: `https://secrets.vaultwares.ca` (tailnet-only).
- Internal service: `http://127.0.0.1:8222` on `vaultwares-1` behind nginx.

What belongs in Vaultwarden:
- VPS access details (SSH keys, break-glass steps)
- CI tokens and API keys
- DB connection strings
- DNS registrar access notes

What should *not* be scattered in docs:
- raw passwords/tokens in plaintext

## Primary: VaultWares Secrets (high priority)

Design requirements (directional):
- Zero-knowledge: the server must not read or reconstruct private/shared decryption keys.
- Tailnet identity binding for access (and later posture checks).
- Audit logs for reads/writes/rotations.
- Rotation workflows for infra secrets and app secrets.

Near-term plan:
1. Define threat model + recovery model.
2. Define client UX (desktop-first) and automation UX (CI integration).
3. Implement storage + crypto envelope + access control.
