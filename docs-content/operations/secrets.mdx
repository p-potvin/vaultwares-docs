# Secrets (Private)

## Primary: VaultWarden (now)

- Purpose: store operational secrets (tokens, DB credentials, renewal notes) **tailnet-first**.
- Location: `greencloud-vps` (`vaultwares-1`) and reachable over Tailnet only.
- Endpoint (canonical): `https://warden.vaultwares.ca` (tailnet-only).
- Legacy URL: `https://secrets.vaultwares.ca` (redirects to `warden.vaultwares.ca`).
- Useful endpoints:
  - `GET /` → service info
  - `GET /health` → `ok`
  - `GET /docs` → FastAPI docs (private)
  - `PUT /v1/secrets/{name}` / `GET /v1/secrets/{name}` → secret CRUD
- Internal service: `http://127.0.0.1:9444` on `vaultwares-1` behind nginx.
- Storage: PostgreSQL (local-only) on `vaultwares-1`.
- Nginx note: `/etc/nginx/nginx.conf` includes **all** files in `/etc/nginx/sites-enabled/*` (not `*.conf`),
  so keep backups in a separate folder (example: `/etc/nginx/backups/`).

What belongs in VaultWarden:
- VPS access details (SSH keys, break-glass steps)
- CI tokens and API keys
- DB connection strings
- DNS registrar access notes

What should *not* be scattered in docs:
- raw passwords/tokens in plaintext

## Design direction

- Tailnet identity binding for access (`tailscale whois`) + server-side allowlists.
- Audit logs for reads/writes.
- Envelope crypto:
  - **Machine secrets**: decryptable on `vaultwares-1` for automation/bots.
  - **User-only secrets**: stored opaque (client-side encryption planned).
