# Webhook secret rotation (`VW_GITHUB_WEBHOOK_SECRET`)

This is the **checklist** for rotating the secret used to verify GitHub webhook signatures.

It must be done **end-to-end**:

- `vw-webhookd` (receiver) must use the new secret
- every GitHub repository webhook pointing to `https://hooks.vaultwares.ca/github` must use the same new secret

If you rotate only one side, you will see `401` or `bad_signature` in logs and deploys will stop.

## Before you start (1 minute)

- Confirm no one is deploying right now.
- Confirm you have tailnet access to `greencloud-vps` (`100.73.93.84`).
- Confirm `https://hooks.vaultwares.ca/health` returns `200`.

## What you rotate (simple definition)

Think of this secret like a **shared “stamp”**:

- GitHub stamps each webhook message with it.
- `vw-webhookd` checks the stamp.
- If the stamp doesn’t match → the message is rejected.

## Checklist (do in this order)

### 1) Generate a new secret

- Generate a new 64-hex value (32 bytes).
- Store it in **VaultWarden** as `VW_GITHUB_WEBHOOK_SECRET`.
- Do not paste the secret into docs, chat, commits, or logs.

### 2) Install the new secret on `greencloud-vps`

- Update the `vw-webhookd` environment file:
  - File: `/etc/vw-webhookd/env`
  - Key: `VW_GITHUB_WEBHOOK_SECRET=...`
- Make sure there is **no leading/trailing whitespace** *and* **no Windows
  line endings**. Saving the file from a Windows editor will silently embed
  `\r` characters that turn every value into garbage (you'll see
  `invalid header field value for "Authorization"` or `bad_signature`).
  Strip CR with `sed -i 's/\r//g' /etc/vw-webhookd/env` right after editing.
- Permissions: `root:vwdeploy 0640` (so the alert scripts and webhookd itself
  can source it, but nothing else on the host can). Historically this file
  was `root:root 0600`, which broke `vw-deploy-notify.sh` running as
  `vwdeploy` — `0640` is the correct mode.

### 3) Restart `vw-webhookd`

- Restart the systemd service: `vw-webhookd.service`
- Verify it is running: `systemctl is-active vw-webhookd` → `active`

### 4) Update GitHub webhooks for every repo

For every repository that deploys via `vw-webhookd`:

- Find the webhook whose URL is `https://hooks.vaultwares.ca/github`
- Update its secret to match the new `VW_GITHUB_WEBHOOK_SECRET`
- Keep events consistent (at minimum: `push`, optionally PR events used by Jira sync)

### 5) Verify (required)

Do **all three**:

1) Ping each repo webhook (GitHub "ping" delivery must be HTTP `200`).
2) Confirm there are **no** `deny: bad_signature` lines in
   `/var/log/vw-webhookd.log` after the rotation.
3) Do one "real" deploy trigger:
   - push a harmless empty commit on one low-risk repo, or
   - run the deploy command manually on `greencloud-vps`.

Then verify in `/var/log/vw-webhookd.log`:

- A `push: repo=…` line for the new delivery.
- A `run: /var/www/deploy-scripts/…sh` line right after.
- `exit=0` ending the dispatch.

If you see `vw_jira_sync_nonblocking_failure` between `run:` and `exit=`,
that's expected when `vw_jira_token` is missing from `vw-secretsd` — the
deploy is **not** blocked by it.

### 6) Verify across orgs (multi-org PATs only)

If the PAT you use elsewhere (e.g. `GH_TOKEN` for `vw-deploy-notify.sh`) is
**fine-grained**, repeat step 5 once per organization (`p-potvin`,
`Prom-King`). A fine-grained PAT scoped to only one org will silently 404
on the others — see `operations/deploy-alerts#gh_token-scope--alerts-repo-choice`.

## Common failure modes (quick diagnosis)

| Symptom | Likely cause | Fix |
|---|---|---|
| GitHub delivery shows `401` | webhook secret mismatch | update repo webhook secret to match `vw-webhookd` |
| `vw-webhookd.log` shows `deny: bad_signature` | secret mismatch, whitespace, **or CRLF** in the env file | `sed -i 's/\r//g' /etc/vw-webhookd/env` then verify no spaces/newlines around the value |
| Deploy runs but fails with `dubious ownership` | git safe.directory missing for the deploy user | add `safe.directory` for that path under the deploy user |
| Deploy runs but can't write to `/var/www/…` | wrong ownership/permissions on the plugin/web dirs | reset to `vwdeploy:www-data 2775` (see `operations/deployment-flow#deploy-script-invariants`) |
| Webhook delivery is signed-OK but deploy never runs (only `vw_jira_sync` lines in log) | unpatched webhookd is short-circuiting on Jira-sync failure | apply the non-blocking patch in `operations/deployment-flow#vw_jira_sync-non-blocking` or set `server.jira_sync.enabled: false` |
| `git fetch origin <SHA>` errors with `not our ref` after the `--all` fetch succeeded | the deploy script still has the legacy by-SHA fetch line | remove it; `--all --prune` already pulls main which contains the SHA |
| `vw-deploy-notify.sh` writes "gh: Not Found (HTTP 404)" for `Prom-King/…` but works for `p-potvin/…` | fine-grained PAT scoped to one org | regenerate the PAT including the other org, or keep alerts in `p-potvin/vaultwares-docs` |

## Notes (ops hygiene)

- `vw-webhookd` runs as user `vwdeploy` on `greencloud-vps`.
- Deploy scripts must be runnable by `vwdeploy` and any target repo checkout + web root must be writable by `vwdeploy`.
- Two service surfaces watch for failures: an on-error notify command on
  the webhookd config and a 5-minute systemd timer (`vw-deny-watch.timer`)
  that consolidates `bad_signature` denies into a single tracking issue.
  See `operations/deploy-alerts`.
