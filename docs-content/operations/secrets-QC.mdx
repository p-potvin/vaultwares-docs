# Secrets (Privé)

## Primaire : VaultWarden (actuellement)

### Résumé simple

Un **secret**, c’est une information que tu ne veux pas afficher publiquement :

- mots de passe
- clés API
- jetons (tokens)
- accès DB

VaultWarden, c’est notre **coffre-fort** pour ces secrets.

### Règle d’or

Si c’est un secret :

- ✅ ça va dans **VaultWarden**
- ❌ ça ne va **pas** dans GitHub, pas dans les docs, pas dans un message Slack/Discord

> Analogie : VaultWarden = le **coffre-fort**. Les docs = le **manuel d’instructions**. On ne met pas l’argent dans le manuel.

### Détails (actuels)

- Objectif : stocker les secrets opérationnels (jetons, identifiants de base de données, notes de renouvellement) **en priorité Tailnet**.
- Emplacement : `greencloud-vps` et accessible uniquement via Tailnet.
- Point de terminaison (canonique) : `https://warden.vaultwares.ca` (tailnet seulement).
- URL héritée : `https://secrets.vaultwares.ca` (redirige vers `warden.vaultwares.ca`).
- Points de terminaison utiles :
  - `GET /` → info du service
  - `GET /health` → `ok`
  - `GET /docs` → docs FastAPI (privé)
  - `PUT /v1/secrets/{name}` / `GET /v1/secrets/{name}` → CRUD de secrets
- Service interne : `http://127.0.0.1:9444` sur `greencloud-vps` derrière nginx.
- Stockage : PostgreSQL (local seulement) sur `greencloud-vps`.
- Note Nginx : `/etc/nginx/nginx.conf` inclut **tous** les fichiers dans `/etc/nginx/sites-enabled/*` (pas `*.conf`),
  donc conservez les sauvegardes dans un dossier séparé (exemple : `/etc/nginx/backups/`).

Ce qui doit être dans VaultWarden :
- Détails d'accès aux VPS (clés SSH, étapes de « break-glass »)
- Jetons CI et clés API
- Chaînes de connexion de base de données
- Notes d'accès au registraire DNS

Ce qui ne devrait *pas* être dispersé dans la documentation :
- mots de passe/jetons bruts en texte clair

## Exemple concret (plus parlant)

| Situation | Mauvaise idée | Bonne idée |
|---|---|---|
| “On doit ajouter une clé API” | Coller la clé dans un fichier `.mdx` | Mettre la clé dans VaultWarden + documenter seulement le **nom** |
| “On doit partager un accès” | Envoyer un mot de passe en message | Partager via VaultWarden (accès contrôlé) |

## Direction de la conception

- Liaison d'identité Tailnet pour l'accès (`tailscale whois`) + listes blanches côté serveur.
- Journaux d'audit pour les lectures/écritures.
- Crypto enveloppe :
  - **Secrets machine** : déchiffrables sur `vaultwares-1` pour l'automatisation/les bots.
  - **Secrets utilisateur uniquement** : stockés de manière opaque (chiffrement côté client prévu).
