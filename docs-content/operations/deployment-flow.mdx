# Deployment flow (Private)

This page is the **source of truth** for how deployments happen for VaultWares + Prom‑King sites.

Related:
- `operations/services-inventory` (single table of all URLs/services)
- `operations/webhook-secret-rotation` (how to rotate `VW_GITHUB_WEBHOOK_SECRET` safely)
- `operations/health-ledger` (active health, routing, alert profile, and Joker model)
- `operations/deploy-alerts` (legacy deploy-alert scripts and replacement notes)

## Principles (mandatory)

- **No GitHub Actions deployments.** A `git push` to `main` must not run build/deploy jobs inside GitHub.
- **GitHub is only an event source.** The only GitHub-side behavior on `main` pushes is a **signed webhook delivery**.
- **Deploy executes inside the tailnet.** Today `greencloud-vps` is the public
  webhook entry point and deployment dispatcher, but production status is defined
  by the project alert profile, not by the host.

## Machines / endpoints

- **Current webhook entry point / dispatcher:** `greencloud-vps` (Public `173.249.194.15`, Tailnet `100.73.93.84`)
- **GitHub webhook receiver (public, required by GitHub):** `https://hooks.vaultwares.ca/github`
  - nginx → `vw-webhookd` on `127.0.0.1:9033`
- **Tailscale webhook handler (tailnet-local):** `vaultwares-hooks` on `127.0.0.1:8787`

## End-to-end flow

```text
Developer pushes to GitHub (main)
  → GitHub delivers signed webhook event (push)
  → https://hooks.vaultwares.ca/github (public)
  → nginx reverse proxy
  → vw-webhookd (127.0.0.1:9033) verifies signature + writes event payload
  → vw-webhookd triggers the deployment runner inside the tailnet
       (via vaultwares-hooks / internal job trigger)
  → runner looks up project routing + alert profile in Health Ledger
  → runner pulls repo + deploys or tests on the configured target host
```

## Repository requirements

- Repos **must not** auto-deploy via `.github/workflows/*` on `push` to `main`.
- PR-only checks are allowed, but any **deployment** must be webhook-driven.
- Each repo must have a GitHub webhook configured to `https://hooks.vaultwares.ca/github`
  (see `operations/project-bootstrap`).

## Secrets / verification

- `VW_GITHUB_WEBHOOK_SECRET` lives in VaultWarden (`https://warden.vaultwares.ca`, tailnet-only) and is provisioned on `greencloud-vps`.
- If you rotate this secret, you must update **both** `vw-webhookd` and every repo webhook. See `operations/webhook-secret-rotation`.
- Verify delivery + processing by checking:
  - `https://hooks.vaultwares.ca/health` returns `200`
  - `vw-webhookd` logs on `greencloud-vps` (and the downstream runner logs)

## vw_jira_sync (non-blocking)

`vw-webhookd` runs the Jira-sync side car (`/usr/local/bin/vw-jira-sync-run.sh`)
in parallel with the deploy command. **The dispatcher is patched so a
`vw_jira_sync` failure does not block the deploy**: failure is logged
(`vw_jira_sync_nonblocking_failure exit=<code>`) and the deploy still runs.
If `vw_jira_token` goes missing from `vw-secretsd`, deploys keep flowing.

If you ever re-roll `/opt/vaultwares-adk/webhookd/vw_webhookd.py`, **re-apply this
patch** around line 295:

```python
if ran_jira and jira_code != 0:
    # jira_sync failures are logged but never block deploy dispatch
    # (the script reaches out to vw-secretsd / Jira, both of which can
    # transiently misbehave for reasons unrelated to the push).
    _append_log(log_path, f"[{_now_iso()}] vw_jira_sync_nonblocking_failure exit={jira_code}")
```

You can also disable `vw_jira_sync` entirely via
`server.jira_sync.enabled: false` in `/etc/vw-webhookd/config.yml`.

## Deploy script invariants (`deploy-tube-sites.sh` and siblings)

These are the boring details that have bitten us when a script gets re-rolled.
Every deploy script that runs as `vwdeploy` must respect them:

| Invariant | Why |
|---|---|
| **No `git fetch origin <SHA>`** | GitHub's HTTPS upload-pack rejects fetching by raw SHA with `not our ref` for any commit not directly advertised. `git fetch --all --prune` already brings in `origin/main`, which contains the SHA. Use `git checkout -f "$SHA"` directly after the `--all` fetch. |
| **Lock file owned by `vwdeploy:vwdeploy`** | `/var/lock/vw-deploy-<repo>.lock` must be writable by `vwdeploy`. If it's left at `root:root` from a prior manual run, deploys exit 1 on `flock`. |
| **`vaultwares-api` lock lives in `/var/lib/vw-deploy/`, mode 0666** | `/var/lock` is `/run/lock` tmpfs (world-writable + sticky). Ubuntu 24.04 ships `fs.protected_regular=2`, which **blocks root from `O_CREAT`-opening a regular file it does not own** in that kind of directory. When `/usr/local/bin/deploy-vaultwares-api.sh` is run manually via `sudo` after a webhook deploy created `vw-deploy-vaultwares-api.lock` as `ubuntu:ubuntu`, root hits `EACCES` on `exec 9>"$LOCK_FILE"`. The script moves the lock to `/var/lib/vw-deploy/vaultwares-api.lock` (non-sticky, not world-writable) and `chmod 0666`s it at the top of every run so whoever invokes (ubuntu via webhook, root via manual sudo) can still open the same file. Don't revert this to `/var/lock`. |
| **`rsync -rltD --delete`, not `rsync -a`** | `-a` (`-rlptgoD`) tries to preserve owner+group from source. As `vwdeploy`, `chgrp` to anything else fails. Use `-rltD` (drop the `-og`). |
| **Plugin dirs owned `vwdeploy:www-data` with `2775`** | rsync writes new files as the running user. Setgid on the dir keeps the group as `www-data` so the web server can read. After perms drift (e.g. someone did `chown -R www-data:www-data`), reset with: `chown -R vwdeploy:www-data <plugin-dir>` + `find <plugin-dir> -type d -exec chmod 2775 {} +` + `find <plugin-dir> -type f -exec chmod 0664 {} +`. |
| **wp-cli wrapped with `sudo -u www-data`** | wp-cli refuses to run as root (the YIKES warning, exit code 1). The deploy script must call `sudo -u www-data wp --path=… rewrite flush --hard \|\| true`. The trailing `\|\| true` tolerates the case where `vwdeploy` cannot sudo (plugins self-flush on version bumps anyway). |
| **GIT_ASKPASS in env** | The git operations inside `runuser -l vwdeploy …` need `GIT_ASKPASS=/var/www/deploy-scripts/git-askpass-vw-gh-token.sh` and `GIT_TERMINAL_PROMPT=0` exported. |
| **`shared-tube` lock / dispatch** | Uses `/var/lib/vw-deploy/shared-tube.lock` with mode `0666` or pre-created with `umask 000` so that both the dispatcher (`vwdeploy`) and remote executing user (`ubuntu` on `vps-ovhcloud` via Tailscale SSH) can read, write, and obtain the flock. |

## Greencloud → OVH SSH dispatch (Tailscale ACL gate)

The dispatcher scripts on `greencloud-vps` (e.g. `/var/www/deploy-scripts/deploy-vaultwares-api.sh`) SSH `ubuntu@100.67.25.118` over the tailnet to run the OVH-side deploy. Tailscale-SSH is enabled on `vps-ovhcloud`, so the connection is gated by the **Tailscale SSH policy**, not OpenSSH `authorized_keys`. The policy `src` list must include `tag:vps-greencloud` (and any other tag from which deploys are dispatched) for `dst: tag:vps-ovhcloud`.

Symptom when missing: dispatcher logs `tailscale: tailnet policy does not permit you to SSH to this node` and exits non-zero; OVH-side deploy log shows nothing for that commit, repo HEAD on OVH stays behind `origin/main`. We have hit this exactly once when only individual personal IPs were in the SSH rule and not the VPS tag.

If you enable session recording for these SSH rules, **provision a `tsrecorder`** node first or set `enforceRecorder: false`. Without a reachable recorder, sessions are killed immediately with *"This SSH session was terminated by Tailscale because the server could not stream session logs to the session recorder."* — including the dispatcher's automated sessions, which silently breaks all deploys.

## Webhook chain end-to-end check

The fastest after-change sanity test: push a no-op commit to a target repo and confirm both the dispatcher log on greencloud and the deploy log on OVH advance to the new SHA.

```sh
# from any tailnet box
ssh root@100.73.93.84 'tail -3 /var/log/vw-deploy/vaultwares-api-dispatch.log'
ssh ubuntu@100.67.25.118 'sudo -n tail -3 /var/log/vw-deploy/vaultwares-api.log'
```

If the dispatcher fires but OVH log doesn't progress, it's almost always one of:
1. Tailscale SSH ACL doesn't include the source tag (see above).
2. The OVH-side script hit the lock-file ownership invariant — see `/var/log/vw-deploy/<repo>.log` for `line N: /var/lib/vw-deploy/<repo>.lock: Permission denied`.
3. The dispatcher SSH session was killed by an over-eager recorder rule.

## Deploy failure alerts

Legacy GitHub-only alert writers were disabled on 2026-06-03 after review. The
replacement direction is Health Ledger plus Alarm Joker: project routing is
separate from severity, and Medium+ alert fanout must go through the Health
Ledger incident lock/dedupe model. See `operations/health-ledger` and
`operations/deploy-alerts`.
