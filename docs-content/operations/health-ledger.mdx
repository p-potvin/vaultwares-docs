# Health Ledger (Private)

Health Ledger is the source of truth for service health evidence and Joker
operations.

The canonical repository is:

- `C:\Users\Administrator\Desktop\Github Repos\health-ledger`
- GitHub: `p-potvin/health-ledger`

Use that repo for:

- Probe Joker configuration and probe results.
- Alarm Joker severity policy, incident state, and notification fanout.
- Token Joker expiry, rotation, and scope-verification notes.
- The first-party health dashboard.
- Medium+ GitHub/Jira issue routing and `@copilot` assignment or mention rules.
- Existing alert audit notes.

## Runner model

Do not infer production status from the host that receives a webhook or runs a
job. Health Ledger keeps these concerns separate:

| Axis | Meaning | Example |
|---|---|---|
| Ingress/control plane | Where the signed GitHub webhook arrives first. | `greencloud-vps` today, because it is the public tailnet entry point. |
| Project routing | Which host/IP should run a probe, build, deploy, or local check. | `vaultwares-api` routes to `vps-ovhcloud` / `100.67.25.118`. |
| Alert profile | What severity and automation are allowed for the project. | Production services can escalate High/Critical; non-production repo CI failures start Low. |
| AI remediation policy | Whether a local model may be used to repair failures. | Local Ollama/Gemma may be used for non-production Low loops, not important production incidents. |

`greencloud-vps` is the current webhook entry point. It is not the definition of
production. `vps-ovhcloud` currently hosts the VaultWares API and its Postgres
database. Future hosts can be added by changing the project routing sheet,
without changing severity policy.

Production webhooks may fan out to `greencloud-vps`, `vps-ovhcloud`,
`Clopeux-Desktop`, or any future registered tailnet server. Non-production
webhooks currently route to `Clopeux-Desktop`, but that is a routing detail only.

## Non-production CI Joker

For projects with no production environment, such as `vault-guardian`, a push or
PR event should trigger a tailnet runner job instead of defaulting to GitHub
Actions. The first version should:

- receive a signed GitHub webhook,
- look up the project route in Health Ledger,
- run the repo's configured build and tests on the target host,
- record success or failure in Health Ledger,
- classify build/test failure as Low unless the project has a production alert
profile,
- allow a bounded local Ollama repair loop only when the project's AI
remediation policy allows it.

The runner must not decide severity from the host. It reads the project route and
alert profile.

## Ollama Environment

Ollama is its own environment because it may move from `Clopeux-Desktop` to a
cloud host later. Monitor it separately from production services, even when it
runs on the same machine.

The Health Ledger monitor should check:

- API availability and version with `GET /api/version`.
- Installed model inventory with `GET /api/tags`.
- Loaded/running model inventory with `GET /api/ps`.
- Hook-scoped job metrics while a CI Joker repair loop is active.
- Host resource impact during model work: CPU, memory, GPU/VRAM when available,
  disk I/O, network I/O, queue depth, request duration, token throughput, and
  failure/timeout counts.

Ollama metrics are not perfect because the process may be used by other tools.
That is acceptable. Mark hook-scoped samples with a run id so the dashboard can
separate "probably caused by this runner" from background usage.

## Probe Locations

- `greencloud-vps`
- `Clopeux-Desktop`
- `vps-ovhcloud`

Probe routing should avoid self-probing for host-owned services. Greencloud
surfaces are checked from `Clopeux-Desktop` and `vps-ovhcloud`; OVH API
surfaces are checked from `greencloud-vps` and `Clopeux-Desktop`. Server nodes
must be allowed to reach one another on `80` and `443`, otherwise the triangle
collapses into false network failures. This gives the monitor a triangle view
instead of letting a host prove only that its own local stack can see itself.

## Current implementation state

As of 2026-06-05:

- `greencloud-vps` runs `health-ledger-probe.service` from `/opt/health-ledger`.
  The service is enabled and writes probe events/rollups under
  `/opt/health-ledger/data`. As of 2026-06-12, it should not be assigned
  Greencloud-hosted services; it is the remote probe leg for OVH and Clopeux
  surfaces.
- `vps-ovhcloud` runs `health-ledger-probe.service` from `/opt/health-ledger`
  with `JOKER_PROBE_LOCATION=vps-ovhcloud`. It is the remote probe leg for
  Greencloud-hosted surfaces.
- `greencloud-vps` runs `health-ledger-dashboard.service` from
  `/opt/health-ledger`. It binds to `127.0.0.1:8790` and reads Health Ledger
  rollups/events only; it does not run probes or append data.
- `greencloud-vps` has `health-ledger-alarm.service` installed from
  `/opt/health-ledger`, but it is intentionally disabled until
  `/etc/health-ledger/alarm.env` contains real operator notification settings.
  The implementation supports Twilio SMS, SMTP email, and optional
  `Clopeux-Desktop` Windows desktop notification. Real sending requires
  `ALARM_JOKER_ALLOW_SEND=1`.
- `Clopeux-Desktop` runs Windows scheduled task `HealthLedgerProbeJoker`, using
  the headless `conhost.exe` wrapper from
  `health-ledger/ops/windows/start-health-ledger-probe.ps1`.
- `Clopeux-Desktop` also runs `HealthLedgerCiJoker`, a separate local webhook
  listener on `127.0.0.1:8788`. It records CI webhook events and applies
  project route/alert-floor policy; bounded Ollama remediation is a later step.
- `vault-guardian` is registered in `health-ledger/services.yaml` as
  `environment: non_production` with `alert_floor: low`.
- Ollama is registered as its own environment and probed from Clopeux with
  `/api/version`, `/api/tags`, and `/api/ps`.
- The VaultWares API has remote probe views from Greencloud and Clopeux, plus
  the direct OVH runtime through `/openapi.json` and `/healthz`. Gateway-only probes use
  `X_VW_GATEWAY_SECRET`; on `greencloud-vps`, the systemd environment file is
  `/etc/health-ledger/gateway.env` and the API nginx vhost sets the matching
  `X-VW-Gateway-Secret` header. Do not copy the secret value into docs, ledger
  events, GitHub, Jira, or assistant chat.
- Legacy `vw-deny-watch.timer` and `vw-token-expiry-watch.timer` are disabled;
  `vw-webhookd` `notifications.on_error_command` is `/bin/true`.
