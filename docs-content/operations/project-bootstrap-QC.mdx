# Démarrage de nouveau projet (Privé)

Cette page est la liste de contrôle par défaut pour l'intégration d'un **nouveau repo/projet VaultWares** dans le modèle tailnet + synchronisation Jira + secrets.

## 0) Décider les identifiants

- **Nom du repo GitHub :** garder le format tiret (exemple : `vault-warden`).
- **Clé de projet Jira :** majuscules, 2 à 10 caractères (exemple : `VWRN`).
- **Propriétaire/org :** par défaut `p-potvin` à moins que le repo ne soit explicitement ailleurs (dans ce cas, définir `repo_owners` dans `vw-jira-sync`).

## 1) Créer le repo GitHub

- Créer le repo sur GitHub (branche par défaut : `main`).
- Ajouter des fichiers d'instructions factices pour diriger les assistants vers le protocole Niveau 1 :

`AGENTS.md` (racine du repo) :
```md
# VaultWares — pre-instructions (repo stub)
This file is intentionally short. It routes work to the company protocol TOC.
Always start at: `C:\Users\Administrator\Desktop\Github Repos\vaultwares-docs\instructions\ROUTER.md`
Execute the ROUTER routine first (always): scan all protocol categories end-to-end, select relevant categories, then open only the selected summaries in category order.
Execute other routines only when relevant (tools/routines). Ledger is always the last step before replying.
Read full notes only when explicitly prompted: `read full notes`
Mandatory ledger (last step before replying): use `C:\Users\Administrator\Desktop\Github Repos\agent-ledger\scripts\record-agent-change.ps1`
```

`CLAUDE.md` (racine du repo) :
```md
# VaultWares — Claude Code pre-instructions (repo stub)
Start at: `C:\Users\Administrator\Desktop\Github Repos\vaultwares-docs\instructions\ROUTER.md`
This file is intentionally short. Do not duplicate company protocols here.
```

## 2) Accès Tailscale + DNS privé

- Le tailnet est le plan de contrôle par défaut : les services privés doivent être uniquement sur le tailnet.
- Préférer le **DNS tailnet** plutôt que les modifications du fichier hôte par appareil.
- Solution de repli (si nécessaire) : le fichier hôte par appareil peut épingler les noms d'hôte privés aux IPs tailnet.

Référence :
- `operations/network-map`
- `operations/tailscale`

## 3) Secrets (VaultWarden)

- Stocker les secrets sur `greencloud-vps` dans **VaultWarden** (`https://warden.vaultwares.ca`, uniquement sur le tailnet).
- Ne pas commiter les secrets dans git. Ne pas coller les valeurs de secrets dans les docs.
- Fournir les secrets aux services via les fichiers d'environnement `/etc/*` sur le VPS (noms seulement dans les docs).

Référence :
- `operations/secrets`

## 4) Projet Jira + intégration vw-jira-sync

Dans `vw-jira-sync/config.yaml` :

1. Ajouter la clé Jira :
```yaml
repo_project_keys:
  my-new-repo: MYNR
```

2. Ajouter le repo à la liste suivie :
```yaml
repos:
  - my-new-repo
```

3. Créer/renommer le fichier de mappage :
- `mapping/my-new-repo.json`

Puis exécuter :
- `python scripts/backfill.py --repo my-new-repo`

Référence :
- `operations/jira-sync`

## 5) Webhook GitHub → hooks.vaultwares.ca

Pour chaque repo :

- Repo → Settings → Webhooks → Add webhook
  - Payload URL : `https://hooks.vaultwares.ca/github`
  - Content type : `application/json`
  - Secret : `VW_GITHUB_WEBHOOK_SECRET` (stocké dans VaultWarden ; fourni sur le VPS)
  - Events :
    - `push`
    - `pull_request`
    - `pull_request_review`
    - `pull_request_review_comment`
    - `issue_comment`

## 6) Placement des bots / automatisation

- Les projets de bots/agents doivent être sous `vaultwares-adk` (et non `automation-suite`).
- Garder le travail de QA/test de bots isolé des dépôts d'infrastructure ; utiliser des projets Jira séparés.

## Liste de vérification

- Services privés :
  - `curl https://docs.vaultwares.ca/` retourne `200` sur le DNS tailnet.
  - `curl https://secrets.vaultwares.ca/health` retourne `200` sur le DNS tailnet.
  - Forcer le chemin de l'IP publique retourne `403` (application uniquement sur le tailnet).
- Synchronisation Jira :
  - Une livraison de webhook de test résulte en la création/mise à jour d'une tâche Jira pour une PR.
