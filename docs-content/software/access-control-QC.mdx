## Aperçu

VaultAccess est la plateforme de contrôle d'accès et de gestion d'identité zéro confiance de VaultWares. Elle applique des politiques de moindre privilège sur l'ensemble de votre infrastructure — de l'authentification des points d'accès à la gestion des accès privilégiés — et s'intègre nativement avec Active Directory, LDAP, SAML 2.0 et les fournisseurs OAuth 2.0.

## Fonctionnalités principales

### Contrôle d'accès basé sur les rôles (RBAC)

VaultAccess fournit un modèle RBAC hiérarchique avec des rôles intégrés et un support complet pour la définition de rôles personnalisés.

```json
{
  "role": {
    "name": "EncryptedStorageAdmin",
    "description": "Administer VaultDrive fleet",
    "permissions": [
      "storage:devices:read",
      "storage:devices:write",
      "storage:policies:read",
      "storage:policies:write",
      "audit:read"
    ],
    "scope": "department:IT",
    "inherits_from": "BaseEmployee"
  }
}
```

```bash
# Create a custom role
vw-admin role create \
  --name "EncryptedStorageAdmin" \
  --permissions storage:devices:read,storage:devices:write,audit:read \
  --scope department:IT

# Assign role to user
vw-admin user assign-role \
  --user alice@example.com \
  --role EncryptedStorageAdmin

# List all roles
vw-admin role list --format table
```

### Contrôle d'accès basé sur les attributs (ABAC)

Les politiques ABAC évaluent plusieurs attributs contextuels au moment de l'accès, permettant un contrôle granulaire au-delà des simples attributions de rôles.

```json
{
  "policy": {
    "name": "HSMAccessPolicy",
    "effect": "allow",
    "principal": {
      "role": ["SecurityOfficer", "CryptoAdmin"]
    },
    "action": ["hsm:sign", "hsm:generate-key"],
    "resource": "arn:vaultwares:hsm:us-east-1:org123:VaultHSM-Prod-*",
    "condition": {
      "StringEquals": {
        "user:department": "Security",
        "request:mfa_authenticated": "true"
      },
      "IpAddress": {
        "request:source_ip": ["10.0.0.0/8", "192.168.1.0/24"]
      },
      "DateGreaterThan": {
        "request:time": "08:00:00"
      },
      "DateLessThan": {
        "request:time": "18:00:00"
      }
    }
  }
}
```

### Architecture zéro confiance

VaultAccess met en œuvre les principes zéro confiance : chaque demande d'accès est vérifiée, peu importe l'emplacement réseau.

<Steps>
  <Step title="Vérification d'identité">
    Chaque utilisateur et appareil doit s'authentifier avec au moins deux facteurs avant que tout accès ne soit accordé. Les certificats, la biométrie et les jetons matériels sont pris en charge.
  </Step>
  <Step title="Vérification de l'état de l'appareil">
    Avant d'accorder l'accès, VaultAccess vérifie que l'appareil demandeur respecte votre niveau de base de sécurité : niveau de correctif du système d'exploitation, statut de chiffrement du disque, présence d'agent EDR et validité du certificat.
  </Step>
  <Step title="Application du moindre privilège">
    Les utilisateurs ne reçoivent que les autorisations nécessaires pour la tâche spécifique. Les jetons d'accès à durée limitée expirent automatiquement; l'accès permanent est minimisé.
  </Step>
  <Step title="Surveillance continue">
    Les sessions d'accès sont surveillées en temps réel. Un comportement anormal (heures inhabituelles, nouvel emplacement, opérations échouées excessives) déclenche une authentification renforcée ou une terminaison de session automatique.
  </Step>
</Steps>

### Authentification multi-facteurs (MFA)

<AccordionGroup>
  <Accordion title="Méthodes MFA prises en charge">
    - **TOTP** (Google Authenticator, Authy, Microsoft Authenticator)
    - **FIDO2 / WebAuthn** clés de sécurité matérielles (YubiKey, terminal biométrique VaultScan)
    - **Notifications push** via l'application mobile VaultWares Authenticator
    - **OTP SMS / Courriel** (pour la compatibilité héritée — non recommandé pour les environnements de haute sécurité)
    - **Carte à puce / PIV** (authentification basée sur certificat X.509)
    - **Biométrique** via les appareils VaultScan
  </Accordion>
  <Accordion title="MFA adaptative">
    L'édition Entreprise prend en charge le MFA adaptatif, qui augmente les exigences d'authentification en fonction des signaux de risque :
    - Première connexion depuis un nouvel appareil ou un nouvel emplacement → clé matérielle requise
    - Accès depuis l'extérieur du réseau d'entreprise → notification push requise
    - Opération privilégiée (génération de clé HSM) → carte à puce requise
  </Accordion>
</AccordionGroup>

```bash
# Configure MFA policy
vw-admin mfa-policy set \
  --require-mfa true \
  --methods totp,fido2,push \
  --grace-period-hours 0 \
  --apply-to-group AllUsers

# Enroll a hardware key for a user
vw-admin mfa enroll-fido2 \
  --user alice@example.com \
  --device "YubiKey 5C"
```

### Authentification unique (SSO)

<Tabs>
  <Tab title="SAML 2.0">
    ```xml
    <!-- SAML SP metadata for VaultAccess -->
    <EntityDescriptor entityID="https://vaultaccess.example.com/saml/sp">
      <SPSSODescriptor>
        <AssertionConsumerService
          Binding="urn:oasis:names:tc:SAML:2.0:bindings:HTTP-POST"
          Location="https://vaultaccess.example.com/saml/acs"
          index="1"/>
        <NameIDFormat>
          urn:oasis:names:tc:SAML:1.1:nameid-format:emailAddress
        </NameIDFormat>
      </SPSSODescriptor>
    </EntityDescriptor>
    ```

    ```bash
    # Configure SAML IdP
    vw-admin sso configure-saml \
      --idp-metadata-url https://idp.example.com/saml/metadata \
      --entity-id https://vaultaccess.example.com/saml/sp \
      --acs-url https://vaultaccess.example.com/saml/acs \
      --sign-requests true
    ```
  </Tab>
  <Tab title="OAuth 2.0 / OIDC">
    ```bash
    # Configure OIDC provider (e.g., Okta, Azure AD, Auth0)
    vw-admin sso configure-oidc \
      --provider okta \
      --issuer https://yourorg.okta.com/oauth2/default \
      --client-id <your-client-id> \
      --client-secret <your-client-secret> \
      --scopes "openid,profile,email,groups"
    ```
  </Tab>
  <Tab title="Active Directory">
    ```bash
    # Join VaultAccess to Active Directory domain
    vw-admin ad join \
      --domain corp.example.com \
      --dc dc01.corp.example.com \
      --username CORP\\svc_vaultaccess \
      --password <service-account-password> \
      --ou "OU=VaultWares,DC=corp,DC=example,DC=com"

    # Sync AD groups to VaultAccess roles
    vw-admin ad sync-groups \
      --map "CN=SecurityTeam,OU=Groups,DC=corp,DC=example,DC=com:SecurityOfficer" \
      --map "CN=ITAdmins,OU=Groups,DC=corp,DC=example,DC=com:EncryptedStorageAdmin" \
      --schedule "0 * * * *"
    ```
  </Tab>
</Tabs>

### Gestion des accès privilégiés (PAM)

Les capacités PAM sont disponibles dans les éditions Professionnelle et Entreprise.

```bash
# Create a privileged session for HSM administration
vw-admin pam request-session \
  --target hsm01.corp.example.com \
  --role SecurityOfficer \
  --justification "Emergency key rotation - Incident INC-20241115" \
  --duration 2h

# Session ID returned: psess_abc123

# All commands in the session are recorded
vw-admin pam connect --session psess_abc123

# Terminate session when done
vw-admin pam end-session --session psess_abc123

# Review session recording
vw-admin pam replay --session psess_abc123
```

## Exemples d'API

```bash
# Create user via API
curl -X POST https://api.vaultwares.com/v1/users \
  -H "X-API-Key: vw_live_1234567890abcdef" \
  -H "Content-Type: application/json" \
  -d '{
    "email": "bob@example.com",
    "first_name": "Bob",
    "last_name": "Smith",
    "roles": ["ITAdmin"],
    "mfa_required": true,
    "department": "IT"
  }'

# Get user access summary
curl https://api.vaultwares.com/v1/users/user_abc123/access-summary \
  -H "X-API-Key: vw_live_1234567890abcdef"

# Revoke all sessions for a user (emergency lockout)
curl -X POST https://api.vaultwares.com/v1/users/user_abc123/revoke-all-sessions \
  -H "X-API-Key: vw_live_1234567890abcdef"
```
